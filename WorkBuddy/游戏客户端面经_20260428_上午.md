## 📝 上期思考题回顾

> **第 11 期思考题**：你的项目同时有 C++（引擎层）和 Lua（逻辑层），某天线上出了一个 crash，堆栈显示崩在 C++ 的 `UObject::ProcessEvent` 里，但你怀疑是 Lua 侧传了一个已经被 GC 回收的 UObject 引用导致的。你会怎么设计一套机制来**预防**这类跨语言悬空引用问题？

### ✅ 参考答案

跨语言悬空引用是 Lua + UE 项目的"经典坑王"，解法需要三道防线配合：

**第一道防线：Lua 侧用弱代理持有 UObject**

不要让 Lua 直接持有 UObject 的裸指针，而是通过一个 **Proxy 表**间接引用：

```lua
-- Lua 侧的安全引用封装
local UObjectProxy = {}
UObjectProxy.__index = function(self, key)
    local raw = rawget(self, "_raw")
    if raw == nil or not raw:IsValid() then
        error("尝试访问已销毁的 UObject: " .. tostring(rawget(self, "_name")))
        return nil
    end
    return raw[key]
end

function WrapUObject(obj)
    return setmetatable({
        _raw = obj,
        _name = obj:GetName()
    }, UObjectProxy)
end
```

核心思路：每次访问 UObject 属性前，先检查 `IsValid()`，把悬空访问变成可控的 Lua 错误而不是 C++ crash。

**第二道防线：C++ 侧注册销毁回调**

```cpp
// C++ 侧：UObject 被 GC 时，通知 Lua 侧清理引用
void UMyObject::BeginDestroy()
{
    Super::BeginDestroy();
    // 通知 Lua 侧：这个对象已经没了
    if (LuaBridge)
    {
        LuaBridge->InvalidateRef(this);  // 将 Proxy._raw 置 nil
    }
}
```

**第三道防线：防御性兜底检查**

在 UnLua 的 `ProcessEvent` 入口处加一层防御：

```cpp
// 调用 Lua 函数前检查 Object 有效性
if (!IsValid(Object))
{
    UE_LOG(LogUnLua, Warning, TEXT("Skipping ProcessEvent: Object is pending kill"));
    return;  // 直接跳过，不传给 Lua
}
```

**三道防线的关系**：

```
Lua 访问 UObject
    │
    ▼
[第1道] Proxy:IsValid() 检查 ──→ Lua error（可 pcall 捕获）
    │
    ▼ （如果第1道被绕过）
[第2道] BeginDestroy 回调 ──→ 主动清 nil
    │
    ▼ （极端情况兜底）
[第3道] ProcessEvent 入口检查 ──→ 跳过调用，打日志
```

面试中答出"三道防线 + Proxy 模式"就很加分了。重点不是记住每一行代码，而是理解**分层防御**的思路。

## 🎯 本期面试题

### Shader 变体（Shader Variants）管理 ⭐⭐ 中级

> **面试题**：你的项目打包后发现 Shader 编译时间很长、包体很大，排查后发现是 Shader 变体数量爆炸。请解释什么是 Shader 变体，为什么会"爆炸"，以及你在项目中是怎么管理和优化的？

#### 📋 参考答案

**什么是 Shader 变体？**

一段 Shader 代码通过 `#ifdef` / `#if` 等预编译指令支持不同的功能开关（比如：是否开阴影、是否开法线贴图、是否开雾效）。编译时，每种开关组合都会生成一份独立的 GPU 程序，这就是一个"变体"。

**为什么会爆炸？**

变体数量 = 所有 keyword 组合的笛卡尔积。假设你有 3 组 keyword，每组 2 个选项：

```
keyword A: ON / OFF          → 2 种
keyword B: SHADOW / NOSHADOW → 2 种  
keyword C: FOG / NOFOG       → 2 种
────────────────────────────
总变体数 = 2 × 2 × 2 = 8 种
```

看起来还好？但实际项目里，一个稍复杂的 Shader 轻松有 10+ 组 keyword，那就是 **2^10 = 1024** 个变体。多个 Shader 叠加后，项目级别的变体总数可能达到**几十万甚至上百万**——这就是"变体爆炸"。

**爆炸的后果**：

| 阶段 | 影响 |
|------|------|
| 打包时 | 编译时间暴涨（可能几小时） |
| 运行时 | 首次使用时编译导致卡顿（"Shader 编译卡顿"） |
| 包体 | Shader 二进制占用大量磁盘和内存 |

## 📖 知识讲解

### 一、UE5 的材质系统与 Shader 变体

UE5 的材质编辑器是"节点式"的，看起来不像在写 Shader，但本质是一样的——每个材质节点最终都会被翻译成 HLSL 代码，然后编译成变体。

**UE5 变体的来源**：

```
┌───────────────────────────────────────┐
│         UE5 Shader 变体来源           │
├───────────────┬───────────────────────┤
│ 材质自身       │ 节点连线的 Feature 开关 │
│ 渲染 Pass      │ BasePass/ShadowPass 等 │  
│ 顶点工厂       │ Static Mesh / Skeletal │
│ 质量等级       │ Low/Medium/High/Epic   │
│ 平台           │ D3D12/Vulkan/Metal 各一套│
└───────────────┴───────────────────────┘
```

**UE5 的管理手段**：

```cpp
// 1. 在 Project Settings → Rendering 中关闭不用的 Feature
// 比如你的项目不用 Forward Shading，就别勾——少一个维度就砍一半变体

// 2. 材质里用 Static Switch（静态开关）而不是动态分支
// Static Switch 在编译期就确定分支，不会产生额外变体
// 只有用到的组合才会编译

// 3. Material Quality Level 限制
// 不要设 4 档质量，很多项目 2 档就够了（Low + High）
```

**UE5 特有的优势：PSO Caching（Pipeline State Object 缓存）**

```cpp
// UE5 的做法：提前收集需要的 PSO，打包到游戏里
// 玩家不会遇到"首次编译卡顿"
// 在 Editor 中：Project Settings → Rendering → "Share Material Shader Code"

// 在命令行收集：
// UnrealEditor-Cmd.exe YourProject -run=DerivedDataCache -fill
```

### 二、Unity 的 Shader 变体管理

Unity 用 `multi_compile` 和 `shader_feature` 两种方式声明 keyword，理解它们的区别是面试高频考点：

```hlsl
// ShaderLab 示例

// multi_compile：所有组合都编译（变体多，但运行时一定有）
#pragma multi_compile _ SHADOW_ON
#pragma multi_compile _ FOG_ON

// shader_feature：只编译材质球实际引用的组合（变体少，但可能漏）
#pragma shader_feature _ _NORMAL_MAP
#pragma shader_feature _ _EMISSION
```

**两者的核心区别**：

```
┌────────────────┬─────────────────┬──────────────────┐
│                │ multi_compile   │ shader_feature   │
├────────────────┼─────────────────┼──────────────────┤
│ 编译哪些变体    │ 所有组合         │ 只编材质球用到的  │
│ 变体数量       │ 多（指数增长）    │ 少               │
│ 运行时切换      │ 可以随时切       │ 没编的就没有      │
│ 适合场景       │ 全局开关（雾、阴影）│ 材质级开关（法线图）│
│ 风险           │ 变体爆炸         │ 漏收集→紫红报错   │
└────────────────┴─────────────────┴──────────────────┘
```

**Unity 变体剔除的实战三板斧**：

```csharp
// 方法1：IPreprocessShaders 接口——打包时自动剔除不要的变体
// 这是最推荐的做法，不依赖人工收集

using UnityEditor.Build;
using UnityEditor.Rendering;

class ShaderVariantStripper : IPreprocessShaders
{
    public int callbackOrder => 0;

    public void OnProcessShader(
        Shader shader, ShaderSnippetData snippet,
        IList<ShaderCompilerData> data)
    {
        for (int i = data.Count - 1; i >= 0; i--)
        {
            // 项目不用 POINT_LIGHT，直接干掉
            if (data[i].shaderKeywordSet.IsEnabled(
                new ShaderKeyword("POINT_COOKIE")))
            {
                data.RemoveAt(i);
            }
        }
    }
}
```

```csharp
// 方法2：ShaderVariantCollection——手动收集需要的变体
// 在 Graphics Settings 里配置 Preloaded Shaders
// 优点：精确控制   缺点：维护成本高，新效果容易漏

// 方法3：项目约定——从源头控制
// - 禁止滥用 multi_compile，能用 shader_feature 就别用前者
// - 全局 keyword 总数控制在 16 个以内
// - 每个 Shader 的 keyword 组不超过 4 组
```

### 三、一个常见 Shader 效果：溶解（Dissolve）

面试中经常被问"你写过哪些 Shader 效果"——溶解效果代码量小、原理清晰、跨引擎通用，是最好的回答：

**原理**：用一张噪声纹理控制每个像素的"消失顺序"，噪声值低于阈值的像素先 clip 掉。

```hlsl
// HLSL（UE5 Custom Node 或 Unity ShaderLab 都适用）

// 输入：
// _NoiseTex  — 噪声纹理（Perlin / Worley 均可）
// _Threshold — 溶解进度 [0, 1]，0=完整，1=全消失
// _EdgeWidth — 边缘发光宽度
// _EdgeColor — 边缘发光颜色

float noise = tex2D(_NoiseTex, uv).r;
float diff = noise - _Threshold;

// 低于阈值的像素直接丢弃
clip(diff);

// 边缘发光：在即将消失的边界加一个颜色
float edge = 1.0 - smoothstep(0, _EdgeWidth, diff);
float3 finalColor = lerp(baseColor, _EdgeColor, edge);

return float4(finalColor, 1.0);
```

**效果示意**：

```
_Threshold = 0.0    _Threshold = 0.5    _Threshold = 1.0
┌──────────┐       ┌──────────┐       ┌──────────┐
│██████████│       │ ██  █ ██ │       │          │
│██████████│  ──→  │██ ░░░ █ │  ──→  │          │
│██████████│       │ █ ░░░░  │       │          │
│██████████│       │  ██ █ ██│       │          │
└──────────┘       └──────────┘       └──────────┘
  完整显示          ░ = 边缘发光          全部消失
```

**面试追问：为什么用 `clip()` 而不是降低 alpha？**

答：`clip()` 是真正丢弃片元（discard），不进入后续的深度写入和混合。如果用 alpha 混合，需要排序、关深度写入，在移动端性能更差。溶解这种"要么有要么没有"的效果，clip 是最佳选择。

## 💡 实战技巧

### 技巧 1：Shader 变体预热（Warmup）

运行时首次使用某个 Shader 变体时，GPU 需要编译它，会造成"卡一下"。大厂的做法是**在 Loading 界面预热**：

```csharp
// Unity：在加载阶段预热所有需要的变体
public class ShaderWarmup : MonoBehaviour
{
    public ShaderVariantCollection svc;
    
    IEnumerator Start()
    {
        // 在 Loading 界面，逐帧预热，不会一次性卡死
        if (svc != null && !svc.isWarmedUp)
        {
            svc.WarmUp();
            yield return null; // 给 GPU 喘口气
        }
    }
}
```

```cpp
// UE5：通过 PSO Precaching 实现类似效果
// 在 DefaultEngine.ini 中：
// [/Script/Engine.RendererSettings]
// r.PSOPrecaching=1
```

### 技巧 2：移动端 Shader 的"省电三原则"

移动端 GPU 的带宽和算力都很紧张，写 Shader 时要注意：

```
① 能用 half 不用 float
   → 移动 GPU 有 half 专用 ALU，速度翻倍

② 能用 tex2Dlod 不用 tex2D（在顶点着色器中）
   → 避免不必要的 mipmap 计算

③ 减少 Dependent Texture Read（依赖纹理读取）
   → 不要 uv = tex2D(noiseTex, uv).rg; color = tex2D(mainTex, uv);
   → 把 UV 偏移放到顶点着色器中计算
```

### 技巧 3：Lua 层控制材质属性的性能注意事项

```lua
-- ❌ 错误写法：每帧 GetDynamicMaterialInstance
function UpdateDissolve_Bad(meshComp, threshold)
    local mat = meshComp:CreateDynamicMaterialInstance(0) -- 每次创建新实例！
    mat:SetScalarParameterValue("Threshold", threshold)
end

-- ✅ 正确写法：缓存 MID，只创建一次
local cachedMID = nil
function UpdateDissolve_Good(meshComp, threshold)
    if not cachedMID then
        cachedMID = meshComp:CreateDynamicMaterialInstance(0)
    end
    cachedMID:SetScalarParameterValue("Threshold", threshold)
end

-- 💡 更进一步：用 Material Parameter Collection（MPC）
-- 全局共享参数，不需要逐实例设置
-- 适合全屏效果（如溶解进度、全局风力）
```
