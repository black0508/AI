## 📝 上期思考题回顾

> **Q：开放世界 500+ 可交互物件的"附近检测"系统，Tick 轮询 vs Trigger Volume vs 空间分区，怎么选？**

### ✅ 参考答案

最优方案是**分层策略**，不要只用一种：

```
┌─────────────────────────────────────────────────┐
│ 层级 1 — 粗筛（空间哈希/网格）                     │
│   将世界划分为 Cell（如 50m×50m），玩家所在及相邻 Cell │
│   内的物件才进入候选集，通常从 500→30                │
├─────────────────────────────────────────────────┤
│ 层级 2 — 精筛（距离排序 + 分帧处理）                │
│   候选集内按距离排序，每帧只检测最近 N 个             │
│   使用 sqrMagnitude 避免开方                       │
├─────────────────────────────────────────────────┤
│ 层级 3 — 触发响应（Trigger Volume 兜底）            │
│   对于高优先级物件（宝箱、NPC），直接挂小 Trigger     │
│   进入时立即响应，不依赖轮询                         │
└─────────────────────────────────────────────────┘
```

**核心思路**：空间哈希做粗筛 → 距离排序分帧精筛 → 高优先物件 Trigger 保底。500+ 物件不可能全部 Tick 检测，但也不能全挂 Trigger（Overlap 事件太多），三层配合才能兼顾性能和响应性。

## 🎯 面试题

### 题目（⭐⭐ 中级）

**在 Unity 游戏项目中，以下代码会造成什么性能问题？如何优化？**

```csharp
// 每帧调用
void Update()
{
    Vector3 dir = (target.position - transform.position).normalized;
    string logMsg = $"Moving towards {target.name}, dir=({dir.x:F2},{dir.y:F2},{dir.z:F2})";
    Debug.Log(logMsg);
    
    // 伤害计算
    object damage = CalculateDamage(); // 返回 int
    int finalDamage = (int)damage;
    ApplyDamage(finalDamage);
}
```

### ✅ 参考答案

这段代码有 **三个** GC Alloc 问题：

| 问题 | 原因 | 每帧堆分配 |
|------|------|-----------|
| `$"..."` 字符串插值 | 每次生成新字符串对象 | ~200B |
| `target.name` | Unity Object.name 每次返回新 string | ~80B |
| `(object)damage` 装箱 | int → object 发生 Boxing | ~20B |

**优化版本**：

```csharp
// 1. 条件编译去掉发行版日志
[System.Diagnostics.Conditional("UNITY_EDITOR")]
void LogMovement(Vector3 dir)
{
    // 只在 Editor 下执行，不产生运行时开销
    Debug.Log($"Moving dir=({dir.x:F2},{dir.y:F2},{dir.z:F2})");
}

// 2. 避免装箱——直接用值类型接收
int damage = CalculateDamage(); // 返回类型改为 int
ApplyDamage(damage);

// 3. 如果必须格式化字符串，用 stackalloc + Span（.NET Standard 2.1+）
void FormatDamage(int dmg, Span<char> buffer)
{
    dmg.TryFormat(buffer, out int written);
    // buffer[..written] 就是格式化结果，零堆分配
}
```

**关键结论**：游戏客户端热路径（Update/FixedUpdate）中，每帧的堆分配 **必须趋近于零**。

## 📖 知识讲解

### 值类型 vs 引用类型——一张表搞清楚

| 维度 | 值类型 (Value Type) | 引用类型 (Reference Type) |
|------|-------------------|------------------------|
| 常见类型 | int, float, bool, Vector3, **struct**, enum | class, string, array, delegate |
| 存储位置 | 栈（局部变量）或嵌入父对象 | 堆（GC 托管） |
| 赋值行为 | **复制整个值** | **复制引用（指针）** |
| 默认值 | 0 / false / 空结构 | null |
| GC 压力 | 无（栈上自动释放） | 有（需要 GC 回收） |
| 判等方式 | 按值比较 | 按引用比较（除非重写） |

### ⚠️ 三个常见误区

**误区 1：值类型一定在栈上？**

❌ 不对。当值类型作为 class 的字段时，它跟着父对象分配在堆上：

```csharp
class Monster  // 堆上
{
    public Vector3 position;  // 跟着 Monster 在堆上，不在栈上！
    public int hp;            // 同上
}
```

**误区 2：struct 比 class 永远快？**

❌ 大 struct（超过 16 字节）频繁传参时反而更慢，因为每次都是值拷贝：

```csharp
// 不推荐——64 字节的 struct 每次传参拷贝 64 字节
struct BigData { public Matrix4x4 mat; } // 64B

// 优化：使用 in 修饰符传递只读引用，避免拷贝
void ProcessData(in BigData data) { /* 零拷贝只读访问 */ }
```

**误区 3：装箱只在显式转 object 时发生？**

❌ 隐式装箱无处不在：

```csharp
int id = 42;
// 触发装箱的典型场景：
Dictionary<int, string>.ContainsKey(id);  // ✅ 不装箱（泛型约束）
string.Format("ID={0}", id);              // ❌ 装箱！参数是 object
Debug.Log("HP: " + hp);                   // ❌ 装箱！int.ToString() 隐式
interface IMovable { void Move(); }
((IMovable)myStruct).Move();              // ❌ 装箱！struct → interface
```

### Span\<T\> 与 stackalloc——零分配的秘密武器

`Span<T>` 是 C# 7.2 引入的**栈上连续内存视图**，配合 `stackalloc` 可实现零 GC 分配：

```csharp
// 传统方式：每次 new byte[]，产生 GC Alloc
byte[] buffer = new byte[256];  // 堆分配，GC 管理

// 高性能方式：栈上分配 + Span 包装
Span<byte> buffer = stackalloc byte[256];  // 栈上，方法结束自动释放
// 使用方式完全相同
buffer[0] = 0xFF;
buffer.Slice(0, 10).Fill(0);
```

**游戏中的典型应用**：

```csharp
// 场景：技能范围内敌人检测，每帧调用
int FindEnemiesInRange(Vector3 center, float radius, Span<int> resultIds)
{
    int count = 0;
    foreach (var enemy in activeEnemies)
    {
        if (count >= resultIds.Length) break;
        float sqrDist = (enemy.Position - center).sqrMagnitude;
        if (sqrDist <= radius * radius)
        {
            resultIds[count++] = enemy.Id;
        }
    }
    return count;
}

// 调用端——零堆分配
void Update()
{
    Span<int> nearby = stackalloc int[32]; // 最多32个，栈上
    int found = FindEnemiesInRange(transform.position, 10f, nearby);
    for (int i = 0; i < found; i++)
    {
        ProcessEnemy(nearby[i]);
    }
}
```

### Unity 中使用 Span 的注意事项

| Unity 版本 | 支持情况 |
|------------|---------|
| 2021.2+ | 支持 .NET Standard 2.1，可用 Span<T> |
| 2020 及以下 | 仅 .NET Standard 2.0，需 NuGet 引入 System.Memory |
| IL2CPP | 完全支持 stackalloc + Span |
| Burst Jobs | 使用 NativeArray 代替（Span 不能跨线程） |

### 跨引擎/跨语言对比

| 概念 | C# (Unity) | C++ (UE5) | Lua |
|------|-----------|-----------|-----|
| 值 vs 引用 | struct vs class | 值语义 vs 指针/引用 | 8 种基础类型值传递，table 引用传递 |
| 避免分配 | stackalloc + Span | 栈对象 / TInlineAllocator | 复用 table、减少临时 string |
| 内存池化 | ArrayPool\<T\>.Shared | TMemStackBase / FMemory | table pool（手动管理） |
| 零 GC 迭代 | foreach struct enumerator | 范围 for / 迭代器 | ipairs（避免 pairs 创建闭包） |

**UE5 对应写法**：

```cpp
// UE5 中类似 stackalloc 的栈上临时数组
void FindNearbyEnemies(const FVector& Center, float Radius)
{
    // TArray 可以用 TInlineAllocator 避免小数组堆分配
    TArray<AActor*, TInlineAllocator<32>> NearbyEnemies;
    
    for (AActor* Enemy : ActiveEnemies)
    {
        if (FVector::DistSquared(Enemy->GetActorLocation(), Center) 
            <= Radius * Radius)
        {
            NearbyEnemies.Add(Enemy);
        }
    }
}
```

**Lua 对应写法**：

```lua
-- Lua 中避免临时 table 的方式：复用全局缓冲 table
local _nearbyBuffer = {}

function FindNearbyEnemies(cx, cy, radius)
    local count = 0
    local r2 = radius * radius
    -- 清空复用（不创建新 table）
    for i = 1, #_nearbyBuffer do _nearbyBuffer[i] = nil end
    
    for _, enemy in ipairs(activeEnemies) do
        local dx, dy = enemy.x - cx, enemy.y - cy
        if dx*dx + dy*dy <= r2 then
            count = count + 1
            _nearbyBuffer[count] = enemy.id
        end
    end
    return _nearbyBuffer, count
end
```

## 💡 实战技巧

### 1. Unity 项目零 GC 检查清单

```
✅ Update/FixedUpdate 里禁止 string 拼接
✅ 用 NonAlloc 版本 API（Physics.RaycastNonAlloc, GetComponentsInChildren<T>(list)）
✅ foreach 只遍历 List<T>，不遍历 IEnumerable（避免装箱迭代器）
✅ 委托缓存：action ??= MyMethod（避免每次 new delegate）
✅ 数值格式化用 TryFormat 替代 ToString()
```

### 2. struct 设计黄金准则

- 大小 ≤ 16 字节（2 个 long）——避免拷贝开销
- 不可变（readonly struct）——避免防御性拷贝
- 语义上表示"值"——如坐标、颜色、时间戳

```csharp
// 推荐写法：readonly struct + 不超过 16B
readonly struct DamageInfo  // 12 bytes
{
    public readonly int Value;
    public readonly int SourceId;
    public readonly DamageType Type; // enum = int, 4B
}
```

### 3. ArrayPool 复用——中大型临时数组的正确姿势

```csharp
// 当数组大小不确定或可能较大时，用 ArrayPool 代替 new
var pool = ArrayPool<RaycastHit>.Shared;
RaycastHit[] results = pool.Rent(64); // 从池中借出

int hitCount = Physics.RaycastNonAlloc(ray, results);
// ... 处理结果 ...

pool.Return(results, clearArray: true); // 用完归还
```
