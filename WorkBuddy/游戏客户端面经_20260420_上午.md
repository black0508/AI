## 📬 上期思考题答案

> 上期问：一个大型 MMO 的背包系统有 200 个格子，每个格子显示图标、数量、品质框、锁定状态。如果用 MVVM 绑定，200 × 4 = 800 个绑定回调，会不会有性能问题？你会怎么优化？

**答案**：会有问题，但不是绑定本身慢——800 个回调的注册和存储开销很小。真正的性能杀手是 **UI 重建（Rebuild）和渲染**。200 个格子全部实例化，光 Draw Call 就顶不住。核心优化思路有三层：

**第一层：虚拟列表（Virtual List）— 最关键**

只创建"可见区域 + 缓冲区"的格子（比如一屏显示 24 个，缓冲 8 个，实际只有 32 个 UI 实例）。滑动时复用格子：

```
背包共 200 格，但屏幕只看得到 24 格：

┌──────────────────────────┐
│ [物品01] [物品02] ... [物品24] │  ← 可见区域（实际创建的 Widget）
└──────────────────────────┘
  ↑ 滑到底部时，顶部的格子"回收"后填到底部，换上新数据

虚拟列表实际创建：32 个格子（24 可见 + 8 缓冲）
绑定回调：32 × 4 = 128 个（而非 800 个）
```

Unity 的 `LoopScrollRect` 和 UE5 的 `ListView` 都内置了虚拟列表支持。

**第二层：脏标记 + 批量刷新**

不要每次数据变化就立即刷 UI，而是打个脏标记，等到下一帧统一刷新：

```csharp
// 标记脏，不立即刷新
vm.Items[slot].IsDirty = true;

// 下一帧统一刷
void LateUpdate()
{
    foreach (var slot in visibleSlots)
        if (slot.IsDirty) { slot.Refresh(); slot.IsDirty = false; }
}
```

一次交易可能改 10 个格子的数据，脏标记能把 10 次刷新合并成 1 次。

**第三层：对象池复用格子 Prefab**

格子的创建销毁用对象池，避免 GC 和 Instantiate 开销。

**面试加分**：如果你能说出"虚拟列表 + 脏标记 + 对象池"三板斧，面试官基本就满意了。

## 🎯 面试题（⭐⭐ 中级）

**Q：状态同步和帧同步分别是什么？它们的核心区别在哪？你做过的项目用的是哪种，为什么这么选？**

### 参考答案

先看一张对比图，秒懂两者的根本差异：

```
┌─ 状态同步（State Sync）──────────────┐
│                                      │
│  客户端A ──操作──→ 服务器 ──状态──→ 客户端B │
│                     ↑                │
│              服务器跑战斗逻辑           │
│              客户端只管"渲染结果"        │
│                                      │
│  服务器说你血量 50，你就是 50           │
└──────────────────────────────────────┘

┌─ 帧同步（Lockstep）─────────────────┐
│                                     │
│  客户端A ──操作──→ 服务器 ──操作──→ 客户端B │
│                     ↑               │
│              服务器只转发操作指令       │
│              每个客户端自己算战斗逻辑   │
│                                     │
│  服务器说"第 100 帧所有人的操作是这些"  │
│  每个客户端拿着操作自己算结果           │
└─────────────────────────────────────┘
```

| 维度 | 状态同步 | 帧同步 |
|------|---------|-------|
| **战斗逻辑在哪跑** | 服务器 | 客户端 |
| **同步的内容** | 状态（位置、血量、Buff…） | 操作指令（移动、释放技能…） |
| **带宽消耗** | 随实体数增长（100 人就很大） | 固定（只传操作，人数影响小） |
| **反作弊** | 天然强（服务器权威） | 天然弱（客户端可篡改计算） |
| **断线重连** | 简单（服务器推当前状态） | 困难（需要从第 1 帧快进重算） |
| **回放系统** | 困难（需要录状态快照） | 简单（录操作就行，文件极小） |
| **确定性要求** | 不需要 | 严格要求（浮点数一致性！） |
| **典型游戏** | MMO、大逃杀、FPS | MOBA、RTS、格斗 |

**一句话总结**：状态同步是"服务器算好了告诉你结果"，帧同步是"大家拿着同样的操作各算各的，结果必须一样"。

## 📖 知识讲解

### 状态同步：服务器说了算

状态同步是目前大型多人游戏的主流方案。UE5 原生就是状态同步架构，它的 **Replication** 系统是业界标杆。

核心原理：服务器是唯一权威（Server Authoritative），客户端发送输入 → 服务器计算 → 服务器把结果同步给所有客户端。

```cpp
// UE5 C++ — 属性同步（最基础的状态同步）

UCLASS()
class AMyCharacter : public ACharacter
{
    GENERATED_BODY()

    // 标记 Replicated → 服务器改了，客户端自动同步
    UPROPERTY(Replicated)
    int32 Health = 100;

    // RepNotify → 值变化时客户端收到回调
    UPROPERTY(ReplicatedUsing = OnRep_Health)
    int32 Armor = 50;

    UFUNCTION()
    void OnRep_Health()
    {
        // 客户端收到新血量后刷新 UI
        UpdateHealthBar(Health);
    }

    // 注册同步属性（必须实现）
    void GetLifetimeReplicatedProps(
        TArray<FLifetimeProperty>& OutProps) const override
    {
        Super::GetLifetimeReplicatedProps(OutProps);
        DOREPLIFETIME(AMyCharacter, Health);
        DOREPLIFETIME(AMyCharacter, Armor);
    }
};
```

UE5 里只要给属性标上 `Replicated` 或 `ReplicatedUsing`，引擎自动处理序列化、差值压缩、网络传输。你几乎不用碰 Socket。

**RPC（远程过程调用）** 是另一个核心机制：

```cpp
// 客户端 → 服务器（请求开枪）
UFUNCTION(Server, Reliable)
void ServerFire(FVector Direction);

// 服务器 → 所有客户端（播放开枪特效）
UFUNCTION(NetMulticast, Unreliable)
void MulticastPlayFireEffect();

// 服务器 → 拥有者客户端（通知弹药不足）
UFUNCTION(Client, Reliable)
void ClientNotifyLowAmmo();
```

三种 RPC 方向记住：`Server`（客户端→服务器）、`Client`（服务器→拥有者）、`NetMulticast`（服务器→所有人）。面试高频考点。

### 帧同步：大家一起算

帧同步的核心是 **确定性**——所有客户端拿到相同的输入，必须算出完全相同的结果。

```csharp
// Unity C# — 帧同步核心循环（简化版）

public class LockstepManager : MonoBehaviour
{
    private int currentFrame = 0;
    private Queue<FrameInput> inputBuffer = new();

    // 固定逻辑帧率：每秒 15 帧（帧同步不用 60 帧）
    private const float LogicFrameInterval = 1f / 15f;
    private float timer = 0;

    void Update()
    {
        timer += Time.deltaTime;
        while (timer >= LogicFrameInterval)
        {
            timer -= LogicFrameInterval;
            if (inputBuffer.TryDequeue(out var frameInput))
            {
                // 所有客户端用相同的输入推进同一帧
                ExecuteLogicFrame(frameInput);
                currentFrame++;
            }
        }
    }

    void ExecuteLogicFrame(FrameInput input)
    {
        // 处理每个玩家的操作
        foreach (var cmd in input.PlayerCommands)
        {
            var player = GetPlayer(cmd.PlayerId);
            player.ProcessCommand(cmd);  // 移动、技能、道具...
        }
        // 推进物理、碰撞、Buff 等逻辑
        PhysicsWorld.Step(LogicFrameInterval);
    }
}
```

帧同步的**致命难题**——**浮点数不确定性**：

```
同样的代码 sin(1.0f)：
  Intel CPU：  0.8414709568023681640625
  ARM CPU：    0.84147095680236816       ← 最后几位不一样！
  
100 帧后，位置差了 0.1
1000 帧后，位置差了 10.0
10000 帧后，完全对不上了
```

所以帧同步项目必须用 **定点数** 替代浮点数来做战斗计算。这是帧同步开发成本高的核心原因。

```csharp
// 定点数示例（用 int64 模拟小数，精度 1/1000）
public struct FixedPoint
{
    private long rawValue;  // 实际值 = rawValue / 1000
    
    public static FixedPoint FromInt(int v) => 
        new() { rawValue = v * 1000L };
    
    public static FixedPoint operator +(FixedPoint a, FixedPoint b) =>
        new() { rawValue = a.rawValue + b.rawValue };
    
    public static FixedPoint operator *(FixedPoint a, FixedPoint b) =>
        new() { rawValue = a.rawValue * b.rawValue / 1000L };
}
```

### Lua 视角：帧同步在手游中的实践

国内很多 MOBA 手游（如《王者荣耀》早期版本）用 Lua 做帧同步的战斗逻辑：

```lua
-- Lua 帧同步框架核心（简化）

local LogicFrame = 0
local FrameInputs = {}  -- 服务器下发的每帧操作

function OnServerFrameData(frame, inputs)
    FrameInputs[frame] = inputs
end

function LogicUpdate()
    local inputs = FrameInputs[LogicFrame]
    if not inputs then return end  -- 还没收到，等待（卡帧）
    
    for _, cmd in ipairs(inputs) do
        local player = PlayerMgr:Get(cmd.pid)
        if cmd.type == "move" then
            -- 注意：这里必须用定点数运算！
            player.pos = FixedPoint.Add(player.pos, cmd.dir)
        elseif cmd.type == "skill" then
            SkillSystem:Cast(player, cmd.skill_id, cmd.target)
        end
    end
    
    LogicFrame = LogicFrame + 1
end
```

Lua 做帧同步的坑：Lua 原生的 `number` 是双精度浮点，**必须用定点数库**或把核心计算下沉到 C++ 层。

## 💡 实战技巧

### 1. 选型决策树

面试被问"你会怎么选"，可以用这个决策流程：

```
你的游戏类型是什么？
│
├── MMO / 大世界 / 100+ 人
│   └── 状态同步（带宽可按需裁剪，反作弊强）
│
├── MOBA / RTS / 5v5~10v10
│   └── 帧同步（带宽固定，回放简单，观战成本低）
│
├── FPS / TPS / 大逃杀
│   └── 状态同步 + 客户端预测（UE5 原生支持）
│
├── 格斗 / ACT / 1v1~2v2
│   └── 帧同步 + Rollback（GGPO 回滚式帧同步）
│
└── 回合制 / 卡牌
    └── 状态同步（最简单，逻辑全在服务器）
```

### 2. 客户端预测 — 状态同步的灵魂

纯状态同步有个体感问题：按了移动，要等服务器回包才能动，延迟感明显。解决方案是**客户端预测（Client-Side Prediction）**：

```
时间线：
  T=0  客户端按下前进，本地先移动（预测）
  T=50ms  请求到达服务器，服务器计算
  T=100ms 服务器回包，客户端对比：
          - 预测对了 → 无事发生，体验丝滑
          - 预测错了 → 回滚到服务器位置（可能会"拉扯"）
```

UE5 的 `CharacterMovementComponent` 内置了这套机制，开箱即用。这也是 UE 做 FPS/TPS 的核心优势。

### 3. 面试加分回答

当面试官问网络同步时，说出这些关键词能加分：

- **客户端预测 + 服务器和解**（Client Prediction + Server Reconciliation）
- **输入延迟 vs 回滚**（Input Delay vs Rollback）—— 帧同步的两种处理延迟的方式
- **网络裁剪 / 相关性**（Relevancy）—— UE5 的 `ReplicationGraph` 可以只同步玩家附近的 Actor
- **快照插值**（Snapshot Interpolation）—— 状态同步中平滑其他玩家运动的标准做法
- **确定性锁步 + 乐观帧**—— 帧同步中不等所有人输入就先推进，收到后校验/回滚

### 4. 常见踩坑

| 坑 | 说明 |
|----|------|
| UE5 忘记 `SetReplicates(true)` | Actor 默认不同步，必须在构造函数里开启 |
| RPC 参数太大 | 单次 RPC 别传大数据，用属性同步代替 |
| `NetMulticast` 标 `Reliable` | 广播用 Reliable 会阻塞，建议 Unreliable |
| 帧同步漏算一帧 | 一帧不一致 = 后面全崩，日志 + Hash 校验是救命稻草 |
| 帧同步用了浮点数 | 哪怕只有一处 `float` 运算，跨平台就炸 |
