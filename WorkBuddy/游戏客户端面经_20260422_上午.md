## 📝 上期思考题回顾

> **第 7 期思考题**：在一个 MMORPG 中，配置表有 5000+ 条装备数据，大部分字段值和默认模板相同。如何用 Lua 元表设计一个"默认值表"方案来节省内存？需要考虑哪些边界情况？

### ✅ 参考答案

核心思路：**只存差异字段，用 `__index` 回落到默认值表**。

```lua
-- 默认值模板
local DefaultEquip = {
    atk = 0,
    def = 0,
    hp = 100,
    crit = 0,
    level_req = 1,
    quality = 1,
    icon = "default_icon",
    desc = "无描述",
}

-- 创建一条装备记录，只传入与默认值不同的字段
local function CreateEquip(overrides)
    return setmetatable(overrides or {}, { __index = DefaultEquip })
end

-- 实际使用：一把剑只需存 3 个字段，其余走默认值
local sword = CreateEquip({ atk = 150, crit = 20, icon = "sword_01" })
print(sword.atk)       --> 150（自身字段）
print(sword.hp)        --> 100（来自默认值表）
print(sword.quality)   --> 1  （来自默认值表）
```

**内存节省效果**：5000 条数据 × 10 字段 = 50000 个字段存储 → 实际只存差异字段，假设平均 3 个差异字段 = 15000 个，**省 70% 内存**。

**边界情况**：

| 场景 | 问题 | 解决方案 |
|------|------|----------|
| 需要将某字段设为 `nil` | `__index` 会回落到默认值，无法区分"没设"和"设为nil" | 用特殊标记值如 `NULL_MARK = {}` 代替 nil |
| 遍历所有字段 | `pairs()` 只遍历自身字段，不会遍历元表 | 写一个 `fullpairs()` 函数合并默认表遍历 |
| 修改默认值表 | 影响所有已创建的记录 | 配置加载完后 `freeze` 默认表（设 `__newindex` 报错） |

## 🎯 本期面试题

### 面试题：请对比 UE5 动画蓝图（AnimBP）和 Unity Animator 的核心架构差异，并说明各自处理"攻击动画打断移动"这类需求的方式。（⭐⭐ 中级）

---

### ✅ 参考答案

先上一张架构对比全景图：

```
┌─────────────────────────────────────────────────┐
│                UE5 动画蓝图 (AnimBP)             │
│                                                   │
│  AnimGraph（节点图）                               │
│  ├── State Machine ─── 各状态节点                  │
│  ├── Blend Space ──── 多维混合                     │
│  ├── Montage Slot ─── 上层动画覆盖                 │
│  └── Layered Blend ── 按骨骼分层                   │
│                                                   │
│  EventGraph（逻辑图）                              │
│  └── 读取游戏变量 → 驱动 AnimGraph                 │
│                                                   │
│  关键：Montage 是独立于状态机的覆盖层              │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│             Unity Animator Controller             │
│                                                   │
│  Layer 0 (Base Layer)                             │
│  ├── Idle ──Trigger──→ Run                        │
│  ├── Run ───Trigger──→ Idle                       │
│  └── Transition: 条件 + 混合时间                   │
│                                                   │
│  Layer 1 (Override / Additive)                    │
│  └── Attack 状态 (上半身覆盖)                      │
│                                                   │
│  Parameter 面板                                    │
│  └── Speed(float), IsAttacking(bool)...           │
│                                                   │
│  关键：一切皆状态，攻击也是状态机里的节点           │
└─────────────────────────────────────────────────┘
```

**核心差异总结**：

| 维度 | UE5 AnimBP | Unity Animator |
|------|-----------|----------------|
| 状态管理 | State Machine + Montage **双轨** | 纯状态机（多 Layer 叠加） |
| 临时动画（攻击、受击） | 用 **Montage** 覆盖到 Slot 上 | 在状态机里加 Attack 状态 + Trigger 转换 |
| 混合方式 | Blend Space（2D）、分层混合、叠加 | Blend Tree、Layer（Override/Additive） |
| 编程接口 | C++ / 蓝图，`PlayMontage()` | C# `SetTrigger()` / `Playable API` |
| 扩展性 | AnimNode 自定义节点 | `Playable API` 自定义混合图 |
| 上手难度 | 较高（概念多） | 较低（可视化状态机直观） |

## 📖 知识讲解

### 1. UE5 的 Montage 思路："状态机管底层，Montage 管表演"

UE5 的精髓在于 **状态机和 Montage 的分离设计**：

- **状态机**负责基础运动（Idle、Walk、Run、Jump）
- **Montage**负责临时的表演动画（攻击、受击、施法、处决动画）

这意味着你不需要在状态机里为每个攻击技能加一个状态节点。攻击动画通过 Montage **覆盖**到指定的 Slot 上，播完自动恢复。

```cpp
// UE5 C++ —— 播放攻击 Montage
void AMyCharacter::PlayAttack()
{
    UAnimInstance* AnimInst = GetMesh()->GetAnimInstance();
    if (AnimInst && AttackMontage)
    {
        // 播放 Montage，0.2 秒混入
        AnimInst->Montage_Play(AttackMontage, 1.0f);

        // 绑定结束回调
        FOnMontageEnded EndDelegate;
        EndDelegate.BindUObject(this, &AMyCharacter::OnAttackEnd);
        AnimInst->Montage_SetEndDelegate(EndDelegate, AttackMontage);
    }
}

void AMyCharacter::OnAttackEnd(UAnimMontage* Montage, bool bInterrupted)
{
    if (bInterrupted)
    {
        UE_LOG(LogTemp, Warning, TEXT("攻击被打断！"));
    }
    // 恢复正常状态
    bIsAttacking = false;
}
```

**Montage 的杀手级特性**：
- **Section 跳转**：一个 Montage 可以分多段（Section），运行时跳转到任意段（实现连击 Combo）
- **Anim Notify**：在时间轴上插入通知点（判定帧、特效触发、音效播放）
- **Blend In/Out**：独立控制混入混出时间
- **多 Slot 并行**：上半身放攻击 Montage，下半身继续跑步状态机

### 2. Unity 的 Animator 思路："一切皆状态"

Unity 的 Animator Controller 用**多层状态机**解决同样的问题：

```csharp
// Unity C# —— 触发攻击动画
public class PlayerAnimController : MonoBehaviour
{
    private Animator _animator;
    private static readonly int AttackTrigger = Animator.StringToHash("Attack");
    private static readonly int SpeedParam = Animator.StringToHash("Speed");

    void Update()
    {
        // 驱动移动混合树
        float speed = _characterController.velocity.magnitude;
        _animator.SetFloat(SpeedParam, speed);

        // 触发攻击
        if (Input.GetMouseButtonDown(0))
        {
            _animator.SetTrigger(AttackTrigger);
        }
    }

    // 通过 Animation Event 回调
    public void OnAttackHitFrame()
    {
        // 在动画指定帧触发伤害判定
        DealDamage();
    }
}
```

Unity 的做法是在 Animator Controller 中：
1. **Base Layer**：`Idle ↔ Run`（用 Speed 参数的 Blend Tree）
2. **Override Layer**：放一个 `Attack` 状态，用 `AvatarMask` 控制只影响上半身
3. 通过 `SetTrigger("Attack")` 触发转换

**Unity 的痛点**：状态多了以后，Animator 面板会变成"蜘蛛网"——几十个状态互相连线，维护噩梦。这也是为什么大型项目会转向 **Playable API**。

### 3. "攻击打断移动"的具体实现对比

| 步骤 | UE5 做法 | Unity 做法 |
|------|---------|-----------|
| ① 触发攻击 | `Montage_Play(AttackMontage)` | `SetTrigger("Attack")` |
| ② 上半身播攻击 | Montage 覆盖到 `UpperBody` Slot | Override Layer + AvatarMask |
| ③ 下半身继续跑 | 状态机照常运行 | Base Layer 不受影响 |
| ④ 判定帧 | Montage 里的 `AnimNotify` | Animation Event 回调 |
| ⑤ 攻击结束恢复 | Montage 自动 BlendOut | Transition 回 Empty State |
| ⑥ 被新动作打断 | `Montage_Stop()` + 播新的 | `ResetTrigger()` + 设新 Trigger |

### 4. Root Motion — 动画驱动位移

Root Motion 是让**动画本身控制角色移动**（而非代码硬算速度），能让动作更自然。

**UE5**：在 Montage/动画资产上勾选 `Enable Root Motion`，AnimBP 的 `Root Motion Mode` 设置为 `Root Motion from Montages Only`（常见选择）。

**Unity**：在 Animator 上勾选 `Apply Root Motion`，然后在脚本里通过 `OnAnimatorMove()` 回调拿到位移量：

```csharp
// Unity —— 自定义 Root Motion 处理
void OnAnimatorMove()
{
    // deltaPosition 是动画这一帧产生的位移
    Vector3 movement = _animator.deltaPosition;
    // 可以在这里做额外处理（如地面检测、重力）
    _characterController.Move(movement + Physics.gravity * Time.deltaTime);
}
```

**面试加分点**：Root Motion 的常见坑是**网络同步**——动画驱动的位移在不同机器上可能因帧率差异产生漂移，所以联网游戏通常只在本地用 Root Motion，服务器用逻辑速度，定期校正。

## 💡 实战技巧

### 技巧 1：UE5 Montage 连击 Combo 套路

```
Montage "ComboAttack"
├── Section: "Attack1"  →  下一段: "Attack2"（按键触发跳转）
├── Section: "Attack2"  →  下一段: "Attack3"
└── Section: "Attack3"  →  结束

// 代码层面
void AMyCharacter::OnComboInput()
{
    UAnimInstance* Anim = GetMesh()->GetAnimInstance();
    if (Anim->Montage_IsPlaying(ComboMontage))
    {
        // 跳转到下一个 Section
        Anim->Montage_JumpToSection(NextSectionName, ComboMontage);
    }
}
```

**要点**：在每个 Section 末尾设一个 "Combo Window" 的 AnimNotifyState，只有在窗口期内按键才跳转到下一段，否则播完自动结束——这就是大家熟悉的 Combo 手感设计。

### 技巧 2：Unity 大型项目用 Playable API 替代 Animator

Animator 状态机在 **50+ 状态**时基本不可维护。很多大厂（米哈游、莉莉丝等）选择 Playable API 自己搭动画混合图：

```csharp
// Unity Playable API 简单示例
var graph = PlayableGraph.Create("MyAnimGraph");
var output = AnimationPlayableOutput.Create(graph, "Anim", animator);

// 创建两个动画片段
var idleClip = AnimationClipPlayable.Create(graph, idleAnimation);
var runClip = AnimationClipPlayable.Create(graph, runAnimation);

// 混合器
var mixer = AnimationMixerPlayable.Create(graph, 2);
graph.Connect(idleClip, 0, mixer, 0);
graph.Connect(runClip, 0, mixer, 1);

// 动态设置权重
mixer.SetInputWeight(0, 1.0f - speed);
mixer.SetInputWeight(1, speed);

output.SetSourcePlayable(mixer);
graph.Play();
```

**Playable API 的优势**：
- 完全代码控制，不依赖可视化状态机
- 可以运行时动态增删动画节点
- 方便实现自定义混合逻辑（如 Motion Matching 的匹配混合）
- 性能更好（跳过 Animator Controller 的状态机 Evaluate 开销）

### 技巧 3：Anim Notify 的正确使用姿势

无论 UE 还是Unity，**动画事件/通知**是连接动画和游戏逻辑的桥梁：

| 用途 | UE5 AnimNotify | Unity Animation Event |
|------|---------------|----------------------|
| 攻击判定帧 | `AnimNotify_DealDamage` | `OnAttackHitFrame()` |
| 脚步声 | `AnimNotify_FootstepSound` | `OnFootStep()` |
| 特效生成 | `AnimNotify_SpawnEffect` | `SpawnVFX()` |
| 窗口期 | `AnimNotifyState`（Start/End） | 用两个 Event 标记开始/结束 |

**避坑**：
- 不要在 Notify 里做重逻辑（如网络请求），Notify 是动画线程回调
- UE5 的 `AnimNotifyState` 有 `Begin/Tick/End` 三个回调，比单点 Notify 更适合做"窗口期"判定
- Unity 的 Animation Event 不支持 `NotifyState` 概念，需要自己用两个 Event 模拟
