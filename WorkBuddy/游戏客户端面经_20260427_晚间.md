## 📝 上期思考题回顾

> **第 10 期思考题**：开放世界场景中，远处物体突然"弹出来"（Pop-in）是常见的体验问题。如果你是客户端负责人，会设计怎样的方案来缓解 Pop-in？

### ✅ 参考答案

Pop-in 的本质是**资源加载速度跟不上玩家移动速度**。解法不是单点优化，而是组合拳：

```
┌─────────────────────────────────────────────────┐
│           Pop-in 缓解的四板斧                     │
├─────────┬───────────────────────────────────────┤
│ 预加载   │ 根据玩家朝向 + 速度矢量，预测 2-3 秒后   │
│         │ 的位置，提前异步加载那个区域的资源          │
├─────────┼───────────────────────────────────────┤
│ LOD 过渡 │ 不要 LOD0→LOD2 直接跳变，中间插一个       │
│         │ CrossFade（Dither 或 Alpha 混合）过渡     │
├─────────┼───────────────────────────────────────┤
│ 优先级   │ 大型建筑/地标 > 中型道具 > 小型装饰        │
│         │ 优先加载视觉冲击大的物体，小物件慢一拍没感觉  │
├─────────┼───────────────────────────────────────┤
│ Impostor │ 远距离用 Billboard/Impostor 代替真模型     │
│         │ UE5 HLOD / Unity HLOD 都有现成方案        │
└─────────┴───────────────────────────────────────┘
```

**UE5 额外方案**：World Partition + Nanite 天然就在做流式加载 + 连续 LOD，Pop-in 问题比传统方案轻很多。但对美术规范要求高——Nanite 对透明材质、Morph Target 不友好。

**Unity 额外方案**：Addressables 异步加载 + `LoadSceneMode.Additive` 分块加载，配合自定义 `StreamingManager` 按优先级队列调度。

核心思路：**不是消灭 Pop-in，而是让它发生在玩家感知不到的地方。**

## 🎯 今日主题：跨语言交互

游戏客户端项目里，纯用一种语言开发几乎不存在。UE 项目标配 C++ + Lua（UnLua/slua），Unity 项目要么 C# + Lua（tolua/xlua），要么走 HybridCLR 做 C# 热更。**跨语言边界是 bug 高发地带，也是面试重灾区。**

## 🎯 面试题 ⭐⭐ 中级

> **面试题**：在 UE 项目中使用 UnLua，当 Lua 侧调用一个 C++ 的 `UFUNCTION` 时，底层经历了哪些关键步骤？如果这个函数有一个 `FVector` 类型的输出参数（`out`），Lua 侧是怎么拿到返回值的？

### ✅ 参考答案

整个调用链可以用一张图概括：

```
Lua 代码                     C++ 引擎
────────                     ────────
obj:DoSomething(pos)
      │
      ▼
[UnLua 注册的 thunk 函数]
      │  ① 从 Lua 栈取参数，转为 FProperty 写入 Params 内存块
      │  ② 标记 out 参数的 CPF_OutParm 标志位
      ▼
UObject::ProcessEvent(UFunction*, Params)
      │
      ▼
[C++ 原生函数执行，写回 out 参数到 Params]
      │
      ▼
[UnLua thunk 函数继续]
      │  ③ 扫描 UFunction 的 PropertyLink
      │  ④ 找到带 CPF_OutParm 的属性，从 Params 读回
      │  ⑤ 推入 Lua 栈作为多返回值
      ▼
local result = obj:DoSomething(pos)  -- Lua 拿到 FVector
```

**关键细节**：
- **FVector 在 Lua 侧是 userdata**，UnLua 为常用的 UE 数学类型注册了专门的 metatable，支持 `+`、`-`、`*` 等运算符重载
- **生命周期**：这个 userdata 的内存由 Lua GC 管理，析构时不需要通知 UE 侧（因为 FVector 是值类型，不是 UObject）
- **性能注意**：频繁跨语言传递 FVector 会产生大量 userdata 分配，高频场景（如 Tick 里每帧算位移）建议缓存 userdata 或用 `x, y, z` 三个 number 传递

## 📖 知识讲解

### 1. 跨语言绑定的三种主流模式

```
┌──────────────┬──────────────────┬──────────────────┐
│    方案       │  UE 生态          │  Unity 生态       │
├──────────────┼──────────────────┼──────────────────┤
│ C++/Lua 绑定  │ UnLua, slua-ue   │ tolua, xlua      │
│ C#/Lua 绑定   │ —                │ xlua, tolua++    │
│ C# 热更      │ —                │ HybridCLR(Huatuo)│
│ Lua 热更     │ UnLua + 文件替换  │ xlua/tolua 热重载 │
└──────────────┴──────────────────┴──────────────────┘
```

### 2. UE + Lua（以 UnLua 为例）

UnLua 的核心设计哲学：**利用 UE 的反射系统，自动把 UFUNCTION/UPROPERTY 暴露给 Lua，不需要手写胶水代码。**

```lua
-- Lua 侧覆写蓝图事件（UnLua 最常见用法）
local M = UnLua.Class()

function M:ReceiveBeginPlay()
    -- 直接调用 C++ 暴露的函数
    local loc = self:K2_GetActorLocation()
    print("Actor 位置:", loc.X, loc.Y, loc.Z)

    -- 调用有 out 参数的函数，返回值自动展开
    local bHit, hitResult = UKismetSystemLibrary.LineTraceSingle(
        self,
        startPos, endPos,
        ETraceTypeQuery.TraceTypeQuery1,
        false, {}, EDrawDebugTrace.None,
        true   -- bIgnoreSelf
    )
    if bHit then
        print("命中:", hitResult.ImpactPoint)
    end
end

return M
```

**绑定原理三步走**：
1. **注册阶段**：UnLua 启动时扫描 UClass 的反射信息，为每个类创建对应的 Lua metatable，UFUNCTION 作为 metatable 的方法
2. **调用阶段**：Lua 调用时触发 thunk 函数，负责 Lua 栈 ↔ UE Params 内存块的数据搬运
3. **覆写阶段**：把原 UFunction 的字节码替换为"调用 Lua 函数"的桥接码，实现 Lua 覆写蓝图事件

### 3. Unity + Lua（以 xlua 为例）

xlua 的绑定思路不同——Unity 没有 UE 那样的运行时反射，所以 xlua 用**代码生成**（Gen）在编译期生成 Wrap 类：

```csharp
// xlua 生成的 Wrap 代码（自动生成，不需要手写）
[LuaCallCSharp]
public class PlayerController : MonoBehaviour
{
    public Vector3 GetPosition() => transform.position;
    public void TakeDamage(float amount) { hp -= amount; }
}
```

```lua
-- Lua 侧使用
local CS = CS  -- C# 命名空间桥
local ctrl = CS.PlayerController.Find("Player")
local pos = ctrl:GetPosition()
ctrl:TakeDamage(10)
```

**xlua vs UnLua 核心区别**：

| 对比项 | UnLua (UE) | xlua (Unity) |
|--------|-----------|-------------|
| 绑定方式 | 运行时反射，自动绑定 | 编译期代码生成（Wrap） |
| 是否需要标注 | UFUNCTION 自动暴露 | 需要 `[LuaCallCSharp]` 标注 |
| 性能热点 | ProcessEvent 的 Params 序列化 | Wrap 函数的装箱/拆箱 |
| 覆写机制 | 替换 UFunction 字节码 | 注入委托到 C# 方法 |

### 4. HybridCLR——Unity 的 C# 原生热更

HybridCLR 的思路很巧妙：**既然 IL2CPP 把 C# 编译成了 C++，那我在 IL2CPP 运行时里加一个 IL 解释器，不就能动态执行新的 C# DLL 了吗？**

```
传统 IL2CPP:
  C# 代码 → IL → IL2CPP 编译 → 原生 C++ → 可执行文件（不能改了）

HybridCLR:
  C# 代码 → IL → IL2CPP 编译 → 原生 C++（AOT 部分，框架层）
                ↘ IL（解释执行） → 可热更新的 DLL（业务层）

  AOT 和解释器共享同一套类型系统、GC、线程模型
```

```csharp
// 热更新加载流程（HybridCLR）
public class HotUpdateLoader : MonoBehaviour
{
    async void Start()
    {
        // 1. 下载最新的热更 DLL
        byte[] dllBytes = await DownloadDLL("HotUpdate.dll");

        // 2. 加载到运行时（这一步是 HybridCLR 的魔法）
        Assembly hotAssembly = Assembly.Load(dllBytes);

        // 3. 反射调用热更代码的入口
        Type entryType = hotAssembly.GetType("HotUpdate.GameEntry");
        entryType.GetMethod("Start")?.Invoke(null, null);
    }
}
```

**HybridCLR vs Lua 热更方案对比**：

| 对比项 | HybridCLR | Lua (xlua/tolua) |
|--------|-----------|-----------------|
| 语言 | 原生 C#，团队零学习成本 | 需要学 Lua，维护两套语言 |
| 性能 | 解释执行约为 AOT 的 1/30~1/50 | Lua 性能类似，但跨语言调用有额外开销 |
| 调试 | 支持断点调试 | 需要额外的 Lua 调试器 |
| 成熟度 | 较新，大型项目案例在增多 | 非常成熟，网易/腾讯大量项目在用 |
| 适用 | 新项目首选，All-in-C# | 老项目改造，或团队有 Lua 积累 |

## 💡 实战技巧

### 技巧一：减少跨语言调用频次

跨语言调用的最大开销不是单次耗时，而是**高频调用的累积**。一个常见的反模式：

```lua
-- ❌ 反模式：每帧在 Lua Tick 里密集调用 C++ 函数
function M:ReceiveTick(DeltaTime)
    local loc = self:K2_GetActorLocation()    -- 跨语言 ×1
    local rot = self:K2_GetActorRotation()    -- 跨语言 ×2
    local vel = self:GetVelocity()            -- 跨语言 ×3
    local hp  = self.CharComp:GetHP()         -- 跨语言 ×4
    -- ... 还有更多
end

-- ✅ 优化：C++ 侧打包成一个函数，一次调用全拿回来
-- C++ 侧：
-- UFUNCTION(BlueprintCallable)
-- void GetFrameSnapshot(FVector& Loc, FRotator& Rot,
--                       FVector& Vel, float& HP);

function M:ReceiveTick(DeltaTime)
    local loc, rot, vel, hp = self:GetFrameSnapshot()  -- 跨语言 ×1
end
```

**经验法则**：Tick 里的跨语言调用控制在 **3-5 次以内**，超过就该考虑打包。

### 技巧二：注意 userdata 的 GC 压力

```lua
-- ❌ 每帧创建临时 FVector userdata
function M:ReceiveTick(dt)
    local offset = UE.FVector(0, 0, 100)  -- 每帧 new 一个
    local target = self:K2_GetActorLocation() + offset
end

-- ✅ 缓存复用
local OFFSET = UE.FVector(0, 0, 100)  -- 模块级缓存
function M:ReceiveTick(dt)
    local target = self:K2_GetActorLocation() + OFFSET
end
```

### 技巧三：热更新架构的边界划分

不管是 Lua 热更还是 HybridCLR，**划好 AOT（不可热更）和 Hot（可热更）的边界**是架构的关键：

```
推荐的分层：
┌─────────────────────────────┐
│  Hot 层（可热更）             │
│  - 游戏逻辑、技能、任务、UI   │ ← Lua / HybridCLR 热更 DLL
│  - 配置表解析、协议处理        │
├─────────────────────────────┤
│  Bridge 层（桥接接口）         │
│  - 定义清晰的 API 契约        │ ← 接口稳定，尽量不改
├─────────────────────────────┤
│  AOT 层（不可热更）           │
│  - 引擎核心、渲染、物理       │ ← C++/C# AOT
│  - 底层框架、SDK 集成         │
└─────────────────────────────┘

原则：Hot 层只依赖 Bridge 层的接口，不直接 include AOT 内部头文件
```
