## 🎯 面试题（⭐⭐ 中级）

**Q：状态机（FSM）和行为树（Behavior Tree）各自的优缺点？什么场景下选哪个？**

### 参考答案

| 维度 | 状态机 FSM | 行为树 BT |
|------|-----------|----------|
| 结构 | 状态 + 转换条件 | 树形节点（选择/序列/条件/动作） |
| 优点 | 直观、调试简单、性能好 | 可复用、易扩展、解耦好 |
| 缺点 | 状态多了转换爆炸（n² 复杂度） | 学习成本稍高、调试需工具 |
| 适合 | 简单 AI（巡逻/追击/攻击 3-5 个状态） | 复杂 AI（Boss 多阶段、策略 NPC） |

**一句话记忆**：3-5 个状态以内用 FSM，超过就上行为树。实际项目经常混合用——行为树的叶子节点里跑小型 FSM。

## 📖 知识讲解

### 状态机 — 简单直接，小怪首选

```cpp
// UE C++ 简单状态机
enum class EAIState { Patrol, Chase, Attack, Flee };

void AEnemyAI::Tick(float DeltaTime)
{
    switch (CurrentState)
    {
    case EAIState::Patrol:
        DoPatrol();
        if (CanSeePlayer())
            CurrentState = EAIState::Chase;
        break;
    case EAIState::Chase:
        MoveToPlayer();
        if (InAttackRange())
            CurrentState = EAIState::Attack;
        else if (!CanSeePlayer())
            CurrentState = EAIState::Patrol;
        break;
    case EAIState::Attack:
        DoAttack();
        if (HP < MaxHP * 0.2f)
            CurrentState = EAIState::Flee;
        break;
    }
}
// 痛点：4 个状态就有 12 种转换，10 个状态就爆炸了
```

Unity 里也一样，只不过通常写在 `MonoBehaviour.Update()` 里，逻辑完全相同。

### 行为树 — 模块化组合，复杂 AI 标配

行为树的核心就两种组合节点：

- **Selector**（选择）：子节点从左到右试，**某个成功就停** → 像 if-elseif
- **Sequence**（序列）：子节点从左到右试，**全成功才算成功** → 像 and 条件链

```
UE5 行为树结构（编辑器里的样子）：

Root
└── Selector（依次尝试，成功就停）
    ├── Sequence [逃跑]
    │   ├── Condition: HP < 20%?
    │   └── Action: FleeFromPlayer
    ├── Sequence [攻击]
    │   ├── Condition: InAttackRange?
    │   └── Action: AttackPlayer
    ├── Sequence [追击]
    │   ├── Condition: CanSeePlayer?
    │   └── Action: MoveToPlayer
    └── Action: Patrol（兜底巡逻）
```

想加新行为？直接挂一个新分支就行，完全不影响其他节点。比 FSM 加状态要优雅太多。

#### Unity 手写行为树核心（精简版）

```csharp
public enum BTStatus { Success, Failure, Running }

public abstract class BTNode
{
    public abstract BTStatus Execute(AIContext ctx);
}

public class Sequence : BTNode
{
    private List<BTNode> children = new();
    public override BTStatus Execute(AIContext ctx)
    {
        foreach (var child in children)
        {
            var s = child.Execute(ctx);
            if (s != BTStatus.Success) return s; // 有一个没成功就停
        }
        return BTStatus.Success;
    }
}

public class Selector : BTNode
{
    private List<BTNode> children = new();
    public override BTStatus Execute(AIContext ctx)
    {
        foreach (var child in children)
        {
            var s = child.Execute(ctx);
            if (s != BTStatus.Failure) return s; // 有一个没失败就停
        }
        return BTStatus.Failure;
    }
}
```

整个行为树框架的核心就这两个类，其余都是往上叠具体的 Condition 节点和 Action 节点。

### 黑板（Blackboard）— 行为树的标配搭档

行为树节点不应该自己去做感知（比如射线检测找敌人），而是从**黑板**上读数据。感知系统（EQS / 自定义 Perception）负责把数据写到黑板上。

```
感知系统 → 写入黑板 → 行为树从黑板读取 → 执行决策
                     （解耦！）
```

UE5 自带 Blackboard 资产，键值对形式存 Actor 引用、位置、布尔值等。Unity 需要自己实现（一个 `Dictionary<string, object>` 就够）。

## 💡 实战技巧

1. **UE5 直接用自带的行为树编辑器**（BehaviorTree + Blackboard + EQS），不用自己造轮子。Task 节点写 C++ 或蓝图都行
2. **Unity 没有官方行为树**，推荐 NodeCanvas（付费插件）或开源的 FluidBehaviorTree
3. **性能注意**：行为树每帧从根节点重评估，NPC 多了有开销 → 降低 AI Tick 频率（比如每 3 帧 Tick 一次），或用事件驱动代替轮询
4. **Lua 项目**（国内手游主流）：通常把行为树配置导出成 JSON/Lua table，运行时解析执行，方便热更
