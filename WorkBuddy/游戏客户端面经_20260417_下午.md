## 📬 上期思考题答案

> 上期问：一个 Boss 有 20 种行为模式，纯行为树会不会也臃肿？有什么更好的组织方式？

**答案**：会的。当行为数量膨胀到 20+，一棵行为树确实会变成"参天大树"——难维护、难调试。业内常见的解法有两条路：

**路线一：分层行为树（Sub-Tree）**

把一棵大树拆成多棵小树，按"阶段"或"行为类别"组织。比如：

```
Boss 主树
├── Phase1_Tree（满血阶段 → 远程 + 近战）
├── Phase2_Tree（半血阶段 → 召唤 + 增强攻击）
└── Phase3_Tree（残血阶段 → 狂暴 + 嘲讽）
```

UE5 原生支持 `Run Sub-Tree` 节点，Unity 用 NodeCanvas 也有 Sub-Graph。阶段切换由一个顶层 Selector 根据 HP 条件决定。每棵子树独立编辑、独立测试，清爽很多。

**路线二：Utility AI（评分决策）**

不走树形结构，而是给每个行为打分——分最高的就执行：

```
远程攻击：距离分 0.8 × 冷却分 0.9 = 0.72
近战攻击：距离分 0.3 × 冷却分 1.0 = 0.30
召唤小怪：血量分 0.6 × 冷却分 0.5 = 0.30
治疗自己：血量分 0.9 × 冷却分 0.8 = 0.72 ← 并列最高，随机选一个
```

Utility AI 的好处是**行为间完全解耦**，加新行为只需定义评分函数，不用动其他逻辑。《杀手》系列和不少 SLG 游戏的 AI 都用这个思路。缺点是调参比较玄学，需要配合可视化调试工具。

**实际工程建议**：大型项目通常混合使用——顶层用 Utility AI 做战略决策（选哪个行为类别），底层用小型行为树执行具体行为序列。

## 🎯 面试题（⭐⭐ 中级）

**Q：游戏客户端的 UI 系统，用 MVC 还是 MVVM？两者在游戏中的核心区别是什么？实际项目中你会怎么选？**

### 参考答案

先上一张对比图：

```
┌─ MVC ─────────────────────┐   ┌─ MVVM ────────────────────────┐
│                           │   │                               │
│  View ←── Controller ──→ Model   View ←═══双向绑定═══→ ViewModel ──→ Model
│   │           ↑           │   │                               │
│   └── 用户操作 ┘           │   │  View 不知道 Model 存在       │
│                           │   │  ViewModel 不知道 View 是谁   │
│  Controller 知道 View     │   │                               │
│  Controller 知道 Model    │   │  靠数据绑定自动同步            │
└───────────────────────────┘   └───────────────────────────────┘
```

| 维度 | MVC | MVVM |
|------|-----|------|
| 数据流 | Controller 手动搬运数据 | 双向绑定自动同步 |
| View 与数据耦合 | 低，但 Controller 会变胖 | 极低，View 只认 ViewModel |
| 适合场景 | 逻辑简单、UI 少的项目 | UI 密集型（RPG、SLG、MMO） |
| 学习成本 | 低 | 中（需要理解绑定机制） |
| 调试难度 | 低（断点打 Controller） | 中（绑定链路较长） |

**一句话总结**：MVC 是"你（Controller）手动搬砖"，MVVM 是"搭好管道让水自己流"。UI 界面多的游戏优先 MVVM，小型项目 MVC 就够了。

## 📖 知识讲解

### 游戏 UI 的痛点

游戏 UI 和 Web 前端的 UI 有个关键差异：**游戏 UI 变化极其频繁**。一个 RPG 的主界面可能同时有：

- 血条/蓝条（每帧刷新）
- 技能 CD（倒计时动画）
- 小地图（实时位置）
- Buff 图标（随时增删）
- 背包/装备（打开关闭切换）
- 弹幕/聊天（异步推送）

如果不做分层，UI 代码很快变成"意大利面条"。

### MVC 在游戏中的经典实现

最朴素的游戏 MVC：

```csharp
// Unity C# — MVC 风格

// === Model：纯数据 ===
public class PlayerData
{
    public int HP;
    public int MaxHP;
    public int Level;
}

// === View：只管显示 ===
public class PlayerHUDView : MonoBehaviour
{
    public Slider hpBar;
    public Text levelText;
    
    public void UpdateHP(int hp, int maxHP)
    {
        hpBar.value = (float)hp / maxHP;
    }
    
    public void UpdateLevel(int level)
    {
        levelText.text = $"Lv.{level}";
    }
}

// === Controller：搬运工 ===
public class PlayerHUDController
{
    private PlayerData model;
    private PlayerHUDView view;
    
    public void OnPlayerDamaged(int damage)
    {
        model.HP -= damage;                     // 改 Model
        view.UpdateHP(model.HP, model.MaxHP);   // 手动刷 View
    }
}
```

**问题暴露**：每次数据变了，都要在 Controller 里手动调 `view.UpdateXXX()`。一个复杂界面有 30 个字段，Controller 就变成 30 个 `Update` 调用的大杂烩——这就是经典的 **Fat Controller** 问题。

### MVVM 的核心：数据绑定

MVVM 的精髓就是：**数据变了，UI 自动刷新；UI 操作了，数据自动更新**。

```csharp
// Unity C# — MVVM 风格（精简实现）

// === ViewModel：可观察的数据层 ===
public class PlayerViewModel
{
    // 可绑定属性：值变化时自动通知
    public BindableProperty<int> HP { get; } = new(100);
    public BindableProperty<int> MaxHP { get; } = new(100);
    public BindableProperty<int> Level { get; } = new(1);
}

// === 核心：BindableProperty（20 行搞定） ===
public class BindableProperty<T>
{
    private T _value;
    public event Action<T> OnValueChanged;
    
    public BindableProperty(T defaultVal) => _value = defaultVal;
    
    public T Value
    {
        get => _value;
        set
        {
            if (!Equals(_value, value))
            {
                _value = value;
                OnValueChanged?.Invoke(value);  // 自动通知！
            }
        }
    }
}

// === View：绑定后就不用管了 ===
public class PlayerHUDView : MonoBehaviour
{
    public Slider hpBar;
    
    public void Bindel(PlayerViewModel vm)
    {
        // 绑定：数据变 → UI 自动刷新
        vm.HP.OnValueChanged += hp =>
            hpBar.value = (float)hp / vm.MaxHP.Value;
    }
}
```

看到区别了吗？View 不需要 Controller 来"推"数据，它自己"订阅"了数据变化。以后不管谁改了 `vm.HP.Value = 50`，血条自动刷新，**改数据的代码完全不需要知道 UI 的存在**。

### UE5 的 MVVM 实践

UE5 从 5.1 起就内置了 **UMG ViewModel** 插件，是 Epic 官方的 MVVM 方案：

```cpp
// UE5 C++ — 使用 UMVVMViewModelBase

UCLASS()
class UPlayerViewModel : public UMVVMViewModelBase
{
    GENERATED_BODY()
    
    // 声明可绑定字段 — UE 用宏自动生成通知逻辑
    UPROPERTY(BlueprintReadWrite, FieldNotify, Setter, Getter)
    int32 HP = 100;
    
    UPROPERTY(BlueprintReadWrite, FieldNotify, Setter, Getter)
    int32 MaxHP = 100;
    
    // Setter 里自动广播变化通知
    void SetHP(int32 NewHP)
    {
        UE_MVVM_SET_PROPERTY_VALUE(HP, NewHP);
    }
    
    int32 GetHP() const { return HP; }
};
```

在 UMG 编辑器里，可以直接把 Widget 的属性（比如 ProgressBar 的 Percent）绑定到 ViewModel 的字段上，**零蓝图代码，零 C++ 胶水**。数据变了 UI 自动刷。

关键点：UE5 MVVM 通过 `MVVMGameSubsystem` 维护全局 ViewModel 列表，任何地方都能访问。

### Lua 项目的 UI 架构（国内手游主流）

国内大部分手游用 Lua 做 UI 逻辑，MVVM 思路一样，只是实现方式不同：

```lua
-- Lua 风格的数据绑定（利用元表 __newindex）

local BindableObject = {}
function BindableObject:New()
    local obj = { _data = {}, _watchers = {} }
    setmetatable(obj, {
        __newindex = function(t, k, v)
            rawset(t._data, k, v)
            -- 赋值时自动触发绑定回调
            if t._watchers[k] then
                for _, cb in ipairs(t._watchers[k]) do
                    cb(v)
                end
            end
        end,
        __index = function(t, k)
            return rawget(t._data, k)
        end
    })
    return obj
end

function BindableObject:Watch(key, callback)
    self._watchers[key] = self._watchers[key] or {}
    table.insert(self._watchers[key], callback)
end

-- 使用
local playerVM = BindableObject:New()
playerVM:Watch("hp", function(val)
    -- UI 自动刷新
    hpBar:SetProgress(val / playerVM.maxHP)
end)

playerVM.hp = 50   -- 赋值即触发 UI 刷新，不用手动调
```

利用 Lua 元表的 `__newindex`，赋值操作本身就是通知。这个方案在腾讯、网易的多个手游项目中广泛使用。

## 💡 实战技巧

### 1. UIManager 是基础设施

不管用 MVC 还是 MVVM，**都需要一个 UIManager 做统一管理**：

```
UIManager 职责清单：
├── 面板的打开/关闭/切换（栈式管理，支持返回）
├── 层级管理（底层HUD → 中层面板 → 顶层弹窗 → 最顶引导）
├── 资源加载（异步加载 Prefab/Widget，配合对象池复用）
├── 生命周期回调（OnOpen/OnClose/OnPause/OnResume）
└── 输入焦点管理（打开背包时屏蔽角色移动输入）
```

### 2. 避免绑定泄漏

MVVM 最容易踩的坑：**View 销毁了但绑定回调还在**，导致空引用崩溃或内存泄漏。

```csharp
// ❌ 错误：View 销毁后回调还在
vm.HP.OnValueChanged += hp => hpBar.value = ...;

// ✅ 正确：View 关闭时解绑
void OnDestroy()
{
    vm.HP.OnValueChanged -= onHPChanged;
}
```

UE5 MVVM 不用担心这个问题——Widget 销毁时引擎自动解绑。

### 3. ViewModel 不要引用 View

这是 MVVM 的**铁律**。ViewModel 只暴露数据和命令（Command），绝不引用具体的 Button、Slider。这样同一个 ViewModel 可以绑定到不同的 View（比如 PC 版和手机版 UI 不同，但数据逻辑共享）。

### 4. 面试加分回答

面试官问 MVC/MVVM 时，如果你能提到：

- "我们项目用 **MVVM + 消息总线** 的组合，ViewModel 之间通过事件通信，避免直接引用"
- "UI 性能优化方面，我们做了 **脏标记 + 批量刷新**，不是每次数据变都立即刷 UI，而是标记为脏，下一帧统一刷新"
- "**对象池复用**常用 UI 元素（如背包格子、聊天气泡），避免频繁创建销毁"

基本上面试官就知道你是有实战经验的。
