## 📝 上期思考题回顾

> **第 8 期思考题**：在一个动作游戏中，角色有"普攻三连"和"技能释放"两套动画。普攻三连需要在特定时间窗口内按键才能接下一段（Combo 系统），而技能释放需要打断当前任何动画并立刻播放。请设计一个**动画优先级系统**。

### ✅ 参考答案

核心思路：**给每类动画定义优先级枚举 + 打断规则表**。

```cpp
// UE5 方案 —— 优先级枚举 + Montage 打断策略
UENUM(BlueprintType)
enum class EAnimPriority : uint8
{
    Idle      = 0,   // 待机/移动
    ComboHit  = 10,  // 普攻连击中的某一段
    Skill     = 50,  // 主动技能
    UltSkill  = 80,  // 大招
    HitReact  = 90,  // 受击硬直（几乎不可打断）
};

// 打断判定：高优先级 > 低优先级直接打断
bool CanInterrupt(EAnimPriority current, EAnimPriority incoming)
{
    if (incoming > current) return true;       // 高优打断低优
    if (incoming == current && current == EAnimPriority::ComboHit)
        return bInComboWindow;                 // 同级Combo仅窗口内可接
    return false;                              // 其余同级/低级不打断
}
```

**UE5 实现要点**：
- 用 `Montage_Play()` 播放，播前调用 `CanInterrupt()` 判断
- Combo 窗口用 `AnimNotifyState` 标记，窗口期内 `bInComboWindow = true`
- 技能释放直接 `Montage_Stop()` 当前 → `Montage_Play()` 技能动画

**Unity 实现要点**：
- 用 Playable API 自建混合图，每个 AnimationClipPlayable 附加 Priority 字段
- 请求播放时对比优先级，高优直接设权重为 1 并 CrossFade
- Combo 窗口用两个 Animation Event 标记 Start/End，中间允许同级接续

**面试加分**：提到"优先级不仅看数值，还要考虑**剩余播放时间**——比如受击动画只剩最后 0.1 秒时，低优先级也应该能接上去"。

## 🎯 面试题

**Q：UE 和 Unity 各自用什么方式管理内存？一个 UObject 和一个 C# GameObject 分别是怎么被回收的？实际项目中最常见的内存泄漏场景有哪些？** ⭐⭐ 中级

### 📋 参考答案

**UE 的内存管理是"双轨制"**：
- **UObject 体系** → 引擎自带的 **标记-清除 GC**（Mark-Sweep），周期性扫描不可达对象
- **非 UObject（原生 C++ 对象）** → 用 **智能指针**（`TSharedPtr` / `TWeakPtr`）或手动 `new/delete`

**Unity 的内存管理也是"双轨制"**：
- **托管对象（C# 堆）** → **Mono/IL2CPP 的分代 GC**，自动回收不可达的托管对象
- **原生资源（Texture/Mesh/材质等）** → 需要手动调用 `Resources.UnloadUnusedAssets()` 或通过 Addressables 释放

**回收过程对比**：

| 维度 | UE (UObject) | Unity (C# 对象) |
|------|-------------|-----------------|
| 回收算法 | 标记-清除（非分代） | 分代 GC（Boehm 或 Incremental） |
| 触发时机 | 引擎每隔 N 帧自动触发 | GC Alloc 超过阈值时自动触发 |
| 暂停 | 分帧标记，卡顿较小 | 可能产生毫秒级 GC Spike |
| 手动控制 | `ForceGarbageCollection()` | `System.GC.Collect()` |
| "弱引用" | `TWeakObjectPtr<>` | `WeakReference<T>` |

**最常见的内存泄漏场景**（面试必答三个）：

1. **UE：忘记设 UPROPERTY()** → C++ 成员指向 UObject 但没标记 `UPROPERTY()`，GC 扫描不到引用，对象被错误回收，后续访问变成野指针（不是泄漏，是更严重的 crash）
2. **UE：循环引用 TSharedPtr** → 两个非 UObject 互相持有 `TSharedPtr`，引用计数永不归零 → 用 `TWeakPtr` 打破
3. **Unity：静态容器/事件残留** → `static List<T>` 或 `static event` 持有对象引用，场景切换后旧对象无法被 GC 回收

## 📖 知识讲解

### 1. UE 的 UObject GC — "标记-清除"的游戏特化版

UE 的 GC 只管 `UObject` 及其子类。核心流程：

```
① 从 GC Root 出发（World、GameInstance、持有 UPROPERTY 的对象）
② 遍历所有可达的 UObject，标记为"存活"
③ 未被标记的 UObject → 调用 BeginDestroy() → 回收内存
```

**关键实战点**：

```cpp
// ❌ 错误：裸指针指向 UObject，GC 看不到这个引用
AActor* CachedActor;  // 没有 UPROPERTY()

// ✅ 正确：用 UPROPERTY() 告诉 GC "我还在用它"
UPROPERTY()
AActor* CachedActor;

// ✅ 也可以用 TWeakObjectPtr（不阻止回收，但访问前能判空）
TWeakObjectPtr<AActor> WeakActor;
if (WeakActor.IsValid())
{
    WeakActor->DoSomething();
}
```

**一句话记忆**：UE 里凡是指向 UObject 的指针，要么加 `UPROPERTY()`，要么用 `TWeakObjectPtr`，裸指针就是定时炸弹。

### 2. UE 的智能指针 — 管"非 UObject"的那一套

UE 对非 UObject 的原生 C++ 对象提供了一套独立的智能指针：

```cpp
// TSharedPtr — 共享所有权，引用计数
TSharedPtr<FMyData> DataA = MakeShared<FMyData>();
TSharedPtr<FMyData> DataB = DataA;  // 引用计数 = 2

// TWeakPtr — 弱引用，不增加引用计数
TWeakPtr<FMyData> WeakData = DataA;
if (TSharedPtr<FMyData> Pinned = WeakData.Pin())
{
    Pinned->Process();  // 安全访问
}

// TUniquePtr — 独占所有权，不可复制
TUniquePtr<FMyData> Unique = MakeUnique<FMyData>();
// TUniquePtr<FMyData> Copy = Unique;  // ❌ 编译错误
TUniquePtr<FMyData> Moved = MoveTemp(Unique);  // ✅ 转移所有权
```

**面试要点**：UE 的 `TSharedPtr` 和标准库的 `std::shared_ptr` 原理一样（引用计数），但 UE 版本**不是线程安全的**（默认 `ESPMode::NotThreadSafe`），多线程场景要用 `ESPMode::ThreadSafe`。

### 3. Unity 的 GC — 为什么"GC Spike"是性能杀手

Unity 用的是 **Boehm GC**（IL2CPP 后端）或 **SGen**（Mono 后端）。核心问题：

```
C# 代码 new 一个对象 → 分配在托管堆上
     ↓
托管堆空间不足 → 触发 GC 暂停
     ↓
GC 扫描所有托管对象，回收不可达的 → 产生 "GC Spike"（卡顿）
```

**Unity 2021+ 的增量式 GC** 可以把扫描分摊到多帧，但并不能完全消除。实战中的核心策略是**减少 GC Alloc，而不是优化 GC 本身**。

常见的 GC Alloc 陷阱和解法：

```csharp
// ❌ 每帧 new 一个临时数组 → 每帧产生 GC Alloc
void Update()
{
    var enemies = new List<Enemy>();  // 每帧分配！
    FindEnemies(enemies);
}

// ✅ 缓存复用 → 0 GC Alloc
private List<Enemy> _enemyCache = new List<Enemy>();
void Update()
{
    _enemyCache.Clear();  // 只清空，不释放内存
    FindEnemies(_enemyCache);
}
```

```csharp
// ❌ 字符串拼接是 GC Alloc 大户
string info = "HP: " + hp + "/" + maxHp;  // 产生多个中间字符串

// ✅ 用 StringBuilder 或 string.Format 缓存
_sb.Clear();
_sb.Append("HP: ").Append(hp).Append("/").Append(maxHp);
string info = _sb.ToString();

// ✅ Unity 2021+ 可以用 Span<char> + stackalloc 做零分配格式化
```

### 4. 对象池 — UE 和 Unity 通用的"GC 减负"方案

对象池的核心思想：**预创建 → 取用 → 回收复用**，避免频繁 new/destroy。

```csharp
// Unity 简易对象池
public class BulletPool : MonoBehaviour
{
    [SerializeField] private GameObject _prefab;
    private Queue<GameObject> _pool = new Queue<GameObject>();

    public GameObject Get()
    {
        var obj = _pool.Count > 0
            ? _pool.Dequeue()
            : Instantiate(_prefab);
        obj.SetActive(true);
        return obj;
    }

    public void Return(GameObject obj)
    {
        obj.SetActive(false);
        _pool.Enqueue(obj);
    }
}
```

```cpp
// UE5 —— 用 FObjectPool 或自定义 TArray 池
// 常见做法：Actor 不 Destroy，而是 SetActorHiddenInGame + Disable
void ABulletPool::ReturnBullet(ABullet* Bullet)
{
    Bullet->SetActorHiddenInGame(true);
    Bullet->SetActorEnableCollision(false);
    Bullet->SetActorTickEnabled(false);
    Pool.Add(Bullet);
}
```

**面试加分**：提到 "UE5 的对象池通常池化 Actor（隐藏+禁用 Tick），而不是真正 Destroy/SpawnActor；Unity 则直接 `SetActive(false)` 放回池中。两者都要注意**回收时重置状态**——比如子弹的速度、特效、碰撞标记，否则复用时带着上一发的残留状态"。

## 💡 实战技巧

### 技巧 1：UE 项目的内存泄漏排查三板斧

```
① Stat MemoryPlatform / Stat Memory   → 整体内存占用概览
② obj list / obj refs                  → 查看特定 UObject 类的实例数量和引用链
③ MemReport -full                      → 输出完整内存报告到日志文件
```

**最高频的泄漏原因**：蓝图或 C++ 中引用了已经不需要的资源（Texture/DataTable 等），导致资源一直不被 GC。用 `obj refs` 查看引用链，找到"谁还在持有它"。

### 技巧 2：Unity 项目的 GC Alloc 清零之路

1. **打开 Profiler → 勾选 GC Alloc 列**，逐帧检查哪些函数有分配
2. **目标**：战斗帧的 GC Alloc 控制在 **0-1 KB**（移动端要求更严格）
3. **必做的几件事**：
   - 容器复用（`List.Clear()` 代替 `new List()`）
   - 避免 `foreach` 对非泛型集合的装箱（Unity 老版本坑）
   - Lambda/闭包如果捕获了外部变量 → 会产生匿名类分配，热路径里改用方法引用
   - `string` 操作全部走 `StringBuilder`

### 技巧 3：Lua 的内存管理注意点

Lua 也有 GC（增量式标记-清除），在 UE+UnLua 项目中需要注意：

```lua
-- ❌ 高频函数里创建临时表 → 每次调用都触发分配
function CalcDamage(atk, def)
    local result = { damage = atk - def, isCrit = false }  -- 临时表
    return result
end

-- ✅ 用多返回值代替临时表 → 0 分配
function CalcDamage(atk, def)
    return atk - def, false  -- damage, isCrit
end
```

**经验法则**：Lua 里 `{}` 和 `..`（字符串拼接）是两大内存分配来源。热路径（每帧调用的函数）尽量不创建新表、不拼接字符串。
