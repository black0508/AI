## 📌 主题一：多线程与异步 — GameThread、RenderThread 与 TaskGraph

---

### 📖 知识讲解

#### 1. UE4 多线程架构总览

UE4 引擎运行时至少包含以下核心线程：

| 线程 | 职责 | 典型任务 |
|------|------|----------|
| **GameThread** | 游戏逻辑主线程 | Tick、输入处理、Actor生命周期 |
| **RenderThread** | 渲染命令生成线程 | 场景代理更新、Draw Call 组织 |
| **RHIThread** | 图形API提交线程 | 将渲染命令提交给 GPU（DX12/Vulkan） |
| **TaskGraph Workers** | 通用工作线程池 | 物理、动画、AI、异步计算 |

**核心设计理念**：GameThread 和 RenderThread **错帧并行**（Frame N 的游戏逻辑与 Frame N-1 的渲染同时进行），通过这种流水线方式大幅提高吞吐量。

```
时间轴:
GameThread:  [Frame 1 逻辑] [Frame 2 逻辑] [Frame 3 逻辑] ...
RenderThread:          [Frame 1 渲染] [Frame 2 渲染] ...
RHIThread:                      [Frame 1 提交] [Frame 2 提交] ...
```

#### 2. GameThread → RenderThread 通信

**绝对禁止**直接从 GameThread 访问渲染线程的数据（如 FSceneProxy）。UE4 提供了 `ENQUEUE_RENDER_COMMAND` 宏来安全地向渲染线程投递任务：

```cpp
// 从 GameThread 向 RenderThread 安全发送命令
ENQUEUE_RENDER_COMMAND(FMyUpdateCommand)(
    [CapturedData = this->SomeData](FRHICommandListImmediate& RHICmdList)
    {
        // 这段代码在 RenderThread 上执行
        // CapturedData 是值拷贝，确保线程安全
        UpdateSceneProxy(CapturedData);
    }
);
```

**关键要点**：
- Lambda 捕获必须是**值拷贝**或确保生命周期安全的共享指针
- `ENQUEUE_RENDER_COMMAND` 是异步的，不会阻塞 GameThread
- 如需同步等待渲染线程完成，使用 `FlushRenderingCommands()`（慎用，会造成卡顿）

#### 3. TaskGraph 系统深入解析

TaskGraph 是 UE4 实现的**有向无环图（DAG）任务调度系统**。与传统线程池的最大区别在于：**TaskGraph 支持声明式的任务依赖关系**。

```cpp
// 定义一个 TaskGraph 任务
class FMyGameTask
{
public:
    FMyGameTask(int32 InData) : Data(InData) {}

    // 任务名称（调试用）
    static const TCHAR* GetTaskName() { return TEXT("FMyGameTask"); }
    FORCEINLINE static TStatId GetStatId() { RETURN_QUICK_DECLARE_CYCLE_STAT(FMyGameTask, STATGROUP_TaskGraphTasks); }

    // 指定在哪个线程执行
    static ENamedThreads::Type GetDesiredThread()
    {
        return ENamedThreads::AnyThread; // 任意工作线程
    }

    // 子任务模式
    static ESubsequentsMode::Type GetSubsequentsMode()
    {
        return ESubsequentsMode::TrackSubsequents; // 跟踪后续依赖
    }

    // 实际执行逻辑
    void DoTask(ENamedThreads::Type CurrentThread, const FGraphEventRef& MyCompletionGraphEvent)
    {
        // 执行计算密集型任务
        ProcessData(Data);
    }

private:
    int32 Data;
};

// 创建并调度任务，支持依赖关系
FGraphEventRef TaskA = TGraphTask<FMyGameTask>::CreateTask(nullptr, ENamedThreads::GameThread)
    .ConstructAndDispatchWhenReady(100);

FGraphEventArray Prerequisites;
Prerequisites.Add(TaskA);

// TaskB 依赖 TaskA 完成后才执行
FGraphEventRef TaskB = TGraphTask<FMyGameTask>::CreateTask(&Prerequisites, ENamedThreads::GameThread)
    .ConstructAndDispatchWhenReady(200);

// 在 GameThread 上等待所有任务完成
FTaskGraphInterface::Get().WaitUntilTaskCompletes(TaskB, ENamedThreads::GameThread);
```

#### 4. TaskGraph vs QueuedThreadPool vs AsyncTask 对比

| 特性 | TaskGraph | QueuedThreadPool | FAsyncTask/FAutoDeleteAsyncTask |
|------|-----------|-------------------|--------------------------------|
| **任务依赖** | ✅ DAG依赖关系 | ❌ 无 | ❌ 无 |
| **线程指定** | ✅ 可指定Named Thread | ❌ 池中任意线程 | ❌ 池中任意线程 |
| **适用场景** | 复杂并行流水线 | 简单并发任务 | 一次性异步操作 |
| **引擎集成** | 深度集成（Physics、Anim等） | 通用用途 | 便捷封装 |
| **开销** | 较低（无锁队列） | 中等 | 中等 |

**选择策略**：
- 任务之间有依赖、需要和引擎子系统协调 → **TaskGraph**
- 简单的"扔出去跑"的异步任务 → **AsyncTask**
- 长时间运行的后台工作（如文件IO） → **QueuedThreadPool** 或 `Async()` 全局函数

#### 5. 实用的 Async() 全局函数（UE4.21+）

对于简单的异步需求，`Async()` 是最简洁的方式：

```cpp
#include "Async/Async.h"

// 在线程池中异步执行
TFuture<int32> Result = Async(EAsyncExecution::ThreadPool, []() -> int32
{
    // 耗时计算
    return HeavyComputation();
});

// 在 TaskGraph 中异步执行
Async(EAsyncExecution::TaskGraph, []()
{
    DoSomethingParallel();
});

// 回到 GameThread 执行回调
Async(EAsyncExecution::TaskGraphMainThread, []()
{
    // 安全更新 UI 或游戏逻辑
    UpdateHUD();
});
```

### 💡 实战技巧

**1. 永远不要在非 GameThread 操作 UObject**

UObject 的 GC 系统不是线程安全的。如果你需要在工作线程中处理数据，先提取为原生类型（FVector、TArray<float> 等），处理完毕后通过 `AsyncTask(ENamedThreads::GameThread, ...)` 回到主线程写回。

```cpp
// ❌ 危险：在工作线程直接操作 UObject
Async(EAsyncExecution::ThreadPool, [Actor = this->MyActor]()
{
    Actor->SetActorLocation(NewLocation); // 可能崩溃！
});

// ✅ 安全：计算后回到 GameThread
Async(EAsyncExecution::ThreadPool, [WeakActor = MakeWeakObjectPtr(MyActor)]()
{
    FVector NewPos = HeavyPathfinding(); // 纯数据计算
    
    AsyncTask(ENamedThreads::GameThread, [WeakActor, NewPos]()
    {
        if (AActor* Actor = WeakActor.Get())
        {
            Actor->SetActorLocation(NewPos); // 在 GameThread 安全操作
        }
    });
});
```

**2. RenderThread 同步陷阱**

`FlushRenderingCommands()` 会让 GameThread 阻塞等待 RenderThread 追上来，**彻底破坏流水线并行**。在性能敏感的场景（如每帧 Tick）中绝对不要使用。只在资源加载/卸载等低频操作中使用。

**3. Lua 侧的异步协作**

在 Lua + UE4 的项目中，Lua 代码跑在 GameThread 上。如果需要做耗时操作（如大量数据排序），可以利用协程分帧处理：

```lua
-- Lua 分帧处理大量数据（协程方式）
function SortLargeTable(data, batchSize)
    batchSize = batchSize or 100
    local sorted = {}
    local index = 1
    
    while index <= #data do
        local batchEnd = math.min(index + batchSize - 1, #data)
        for i = index, batchEnd do
            -- 处理一批数据
            table.insert(sorted, ProcessItem(data[i]))
        end
        index = batchEnd + 1
        
        -- 让出当前帧，下一帧继续
        coroutine.yield()
    end
    
    table.sort(sorted, function(a, b) return a.score > b.score end)
    return sorted
end
```

**4. 性能剖析关键指标**

使用 `stat threading` 和 `stat taskgraph` 控制台命令可以查看多线程运行状态。关注以下指标：
- **GameThread time > 16.6ms**：游戏逻辑卡了，检查 Tick 逻辑
- **RenderThread time > 16.6ms**：渲染提交卡了，检查 Draw Call 数量
- **GT/RT gap**：两个线程之间的等待时间，越大说明流水线越不平衡

### 📖 知识讲解

#### 1. 对象池模式（Object Pool Pattern）

**核心思想**：预先创建一批对象放入池中，需要时取出使用，用完归还而不是销毁。避免频繁的 `new`/`delete` 带来的：
- 内存分配开销（系统调用、锁竞争）
- 内存碎片化
- 构造/析构函数的开销
- GC 压力（在有 GC 的引擎中）

**游戏中的经典场景**：子弹、粒子、音效、伤害数字飘字、小兵等需要频繁创建销毁的对象。

```cpp
// === C++ 通用对象池实现 ===
template<typename T>
class TObjectPool
{
public:
    TObjectPool(int32 InitialSize = 32)
    {
        Expand(InitialSize);
    }

    ~TObjectPool()
    {
        for (T* Obj : AllObjects)
        {
            delete Obj;
        }
    }

    // 从池中获取对象
    T* Acquire()
    {
        if (AvailableObjects.Num() == 0)
        {
            // 池耗尽，扩容
            Expand(AllObjects.Num()); // 翻倍扩容
            UE_LOG(LogTemp, Warning, TEXT("ObjectPool expanded to %d"), AllObjects.Num());
        }

        T* Obj = AvailableObjects.Pop();
        Obj->OnAcquire(); // 激活回调
        ActiveCount++;
        return Obj;
    }

    // 归还对象到池中
    void Release(T* Obj)
    {
        if (!Obj) return;
        
        Obj->OnRelease(); // 重置回调
        AvailableObjects.Push(Obj);
        ActiveCount--;
    }

    int32 GetActiveCount() const { return ActiveCount; }
    int32 GetPoolSize() const { return AllObjects.Num(); }

private:
    void Expand(int32 Count)
    {
        for (int32 i = 0; i < Count; ++i)
        {
            T* NewObj = new T();
            AllObjects.Add(NewObj);
            AvailableObjects.Push(NewObj);
        }
    }

    TArray<T*> AllObjects;         // 所有已分配的对象（用于最终释放）
    TArray<T*> AvailableObjects;   // 可用对象栈
    int32 ActiveCount = 0;
};

// === 子弹类示例 ===
class FBullet
{
public:
    FVector Position;
    FVector Velocity;
    float LifeTime;
    bool bActive;

    void OnAcquire()
    {
        bActive = true;
        LifeTime = 0.f;
        // 不需要重新分配内存，只需重置状态
    }

    void OnRelease()
    {
        bActive = false;
        Position = FVector::ZeroVector;
        Velocity = FVector::ZeroVector;
    }
};

// === 使用方式 ===
TObjectPool<FBullet> BulletPool(256); // 预分配 256 颗子弹

void FireBullet(FVector SpawnPos, FVector Direction)
{
    FBullet* Bullet = BulletPool.Acquire();
    Bullet->Position = SpawnPos;
    Bullet->Velocity = Direction * BulletSpeed;
}

void OnBulletExpired(FBullet* Bullet)
{
    BulletPool.Release(Bullet); // 归还，不销毁
}
```

#### 2. UE4 中的对象池实践

UE4 自带了 `FPooledObject` 思想的应用，但更实用的是 **Actor Pool**：

```cpp
// UE4 Actor 对象池（基于 SetActorHiddenInGame + 禁用碰撞/Tick 实现）
UCLASS()
class UActorPoolSubsystem : public UWorldSubsystem
{
    GENERATED_BODY()
    
public:
    AActor* SpawnFromPool(TSubclassOf<AActor> ActorClass, const FTransform& Transform)
    {
        TArray<AActor*>& Pool = PoolMap.FindOrAdd(ActorClass);
        
        AActor* Actor = nullptr;
        if (Pool.Num() > 0)
        {
            // 从池中复用
            Actor = Pool.Pop();
            Actor->SetActorTransform(Transform);
            ActivateActor(Actor);
        }
        else
        {
            // 池空了，新建
            FActorSpawnParameters Params;
            Params.SpawnCollisionHandlingOverride = ESpawnActorCollisionHandlingMethod::AlwaysSpawn;
            Actor = GetWorld()->SpawnActor(ActorClass, &Transform, Params);
        }
        return Actor;
    }
    
    void ReturnToPool(AActor* Actor)
    {
        DeactivateActor(Actor);
        PoolMap.FindOrAdd(Actor->GetClass()).Add(Actor);
    }

private:
    void ActivateActor(AActor* Actor)
    {
        Actor->SetActorHiddenInGame(false);
        Actor->SetActorEnableCollision(true);
        Actor->SetActorTickEnabled(true);
        // 调用自定义重置接口
        if (auto* Poolable = Cast<IPoolableInterface>(Actor))
        {
            Poolable->OnPoolAcquire();
        }
    }
    
    void DeactivateActor(AActor* Actor)
    {
        Actor->SetActorHiddenInGame(true);
        Actor->SetActorEnableCollision(false);
        Actor->SetActorTickEnabled(false);
        if (auto* Poolable = Cast<IPoolableInterface>(Actor))
        {
            Poolable->OnPoolRelease();
        }
    }

    TMap<UClass*, TArray<AActor*>> PoolMap;
};
```

#### 3. 组件模式（Component Pattern）与 UE4 Actor-Component 架构

**传统继承的问题（菱形继承 / 功能爆炸）**：

```
               GameObject
              /    |     \
        Movable  Renderable  Collidable
           \       |        /
            \      |       /
             Character  ???
```

当你需要一个「可移动 + 可渲染 + 可碰撞 + 可交互 + 有AI」的角色时，继承链变得不可维护。

**组件模式的解法**：将功能拆分为独立的组件（Component），通过组合（Composition）而非继承来构建实体。

```
Actor (空壳容器)
 ├── UStaticMeshComponent (渲染)
 ├── UCapsuleComponent (碰撞)
 ├── UCharacterMovementComponent (移动)
 ├── UMyInteractionComponent (交互)
 └── UAIBrainComponent (AI)
```

**这就是经典的"组合优于继承"（Composition over Inheritance）原则的工程实践。**

#### 5. Lua 侧的组件模式实践

```lua
-- Lua 版轻量级 Entity-Component 框架
local Entity = {}
Entity.__index = Entity

function Entity.new(name)
    return setmetatable({
        name = name,
        components = {},
    }, Entity)
end

function Entity:AddComponent(compType, comp)
    self.components[compType] = comp
    comp.owner = self
    if comp.OnAttach then comp:OnAttach() end
    return self -- 支持链式调用
end

function Entity:GetComponent(compType)
    return self.components[compType]
end

function Entity:RemoveComponent(compType)
    local comp = self.components[compType]
    if comp then
        if comp.OnDetach then comp:OnDetach() end
        comp.owner = nil
        self.components[compType] = nil
    end
end

function Entity:Tick(deltaTime)
    for _, comp in pairs(self.components) do
        if comp.Tick then comp:Tick(deltaTime) end
    end
end

-- 使用示例
local HealthComp = {
    maxHP = 100, currentHP = 100,
    TakeDamage = function(self, dmg)
        self.currentHP = math.max(0, self.currentHP - dmg)
        print(string.format("[%s] HP: %d/%d", self.owner.name, self.currentHP, self.maxHP))
        if self.currentHP <= 0 then
            -- 通过 owner 通知其他组件（简化版事件）
            local deathFX = self.owner:GetComponent("DeathEffect")
            if deathFX then deathFX:Play() end
        end
    end
}

local player = Entity.new("Player")
    :AddComponent("Health", HealthComp)
    :AddComponent("Movement", { speed = 300, Tick = function(self, dt) end })
    :AddComponent("DeathEffect", { Play = function(self) print("Death VFX!") end })
```

### 💡 实战技巧

**1. 对象池的容量策略**

不要盲目设置巨大的初始池。推荐做法：
- 开发阶段：用较小的池 + 扩容日志，统计实际峰值
- 发布阶段：根据统计数据设置 InitialSize = 峰值 × 1.2
- 设置上限：防止异常情况导致无限扩容吃掉内存

```cpp
T* Acquire()
{
    if (AvailableObjects.Num() == 0)
    {
        if (AllObjects.Num() >= MaxPoolSize)
        {
            UE_LOG(LogTemp, Error, TEXT("Pool exhausted! Max size: %d"), MaxPoolSize);
            return nullptr; // 或者强制回收最老的活跃对象
        }
        Expand(FMath::Min(GrowthSize, MaxPoolSize - AllObjects.Num()));
    }
    // ...
}
```

**2. 对象池与网络同步的坑**

在多人游戏中使用 Actor 对象池要特别小心 Replication：
- 归还到池中的 Actor 不应该继续同步给客户端
- 复用 Actor 时需要清除旧的 ReplicatedUsing 状态
- 建议在 `DeactivateActor` 时调用 `SetReplicates(false)`

**3. 组件模式的性能陷阱**

组件越多，每帧 Tick 的开销越大。务必：
- 不需要 Tick 的组件设置 `PrimaryComponentTick.bCanEverTick = false`
- 使用 `SetComponentTickEnabled(false)` 动态关闭不活跃组件的 Tick
- 考虑用事件驱动替代轮询

**4. 在 Lua 热更新项目中的组件管理**

如果用 UnLua 或 sluaunreal，组件的逻辑可以放在 Lua 侧实现，方便热更新。但要注意：
- C++ Component 作为数据容器和引擎接口
- Lua 侧实现业务逻辑，通过绑定函数操作 C++ 组件
- 避免在 Lua 侧持有 UObject 的强引用（GC 问题）

> 2. 千级子弹的碰撞检测如何避免 O(n²) 的暴力遍历？
> 3. 如何利用 GPU Instance 实现上千颗相同材质子弹的高效渲染？
> 4. 这套方案在移动端需要做哪些取舍？
