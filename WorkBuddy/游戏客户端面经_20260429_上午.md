## 📝 上期思考题回顾

> **第 12 期思考题**：你的项目需要实现一个天气系统，下雨时角色身上会逐渐变湿、下雪时角色肩头会积雪。你会如何设计这套"天气影响角色材质"的方案？需要同时考虑性能（同屏 20+ 角色）和美术可控性。

### ✅ 参考答案

这道题考的是**全局参数驱动材质变化**的架构设计，核心是"一处修改、全局生效"。

**第一层：全局参数广播**

用引擎的全局参数机制驱动天气状态，避免逐角色设置：

```cpp
// UE5：Material Parameter Collection (MPC) 方案
// 天气系统只需更新一个 MPC，所有角色材质自动响应
void AWeatherSystem::UpdateWeatherParams(float DeltaTime)
{
    CurrentWetness = FMath::FInterpTo(CurrentWetness, TargetWetness, DeltaTime, 2.0f);
    WeatherMPC->SetScalarParameterValue("Wetness", CurrentWetness);    // 0~1 湿润度
    WeatherMPC->SetScalarParameterValue("SnowCoverage", SnowAmount);   // 0~1 积雪量
}
```

```csharp
// Unity：Shader.SetGlobalFloat 方案（等价思路）
Shader.SetGlobalFloat("_GlobalWetness", currentWetness);
Shader.SetGlobalFloat("_GlobalSnowCoverage", snowAmount);
```

**第二层：材质中响应全局参数**

关键技巧——**不加新 Pass，在现有材质里用分支响应**：

- **湿润效果**：提升 Roughness 下限 + 加深 BaseColor + 法线扰动模拟水膜
- **积雪效果**：在**顶点着色器**中用 WorldNormal.Z 判断朝上面，叠加白色
- **性能控制**：用 `[branch]` 而非 `[flatten]`，Wetness == 0 时跳过计算

**第三层：20+ 角色的性能保障**

- 全局参数只设一次，不是逐角色设 Material Instance Dynamic
- 积雪/湿润变化用 `FInterpTo` 平滑过渡，不要每帧 if-else 跳变
- 远处角色（LOD2+）可跳过积雪的顶点偏移计算，只做颜色叠加

> 💡 **面试加分点**：提到 MPC 是 GPU 端的全局 Uniform Buffer，更新零开销；如果还能提到 Keyword 策略（雨/雪/正常三套 Keyword，运行时切换），说明对变体管理和性能的理解很到位。

## 🎯 面试题

### 题目（⭐⭐ 中级）

> **你的游戏有一个技能系统，战士的"横扫"技能需要检测前方扇形区域内的所有敌人。你会选择什么碰撞检测方案？如果同屏有 50 个敌人、每帧有 10 个技能同时释放，你会怎么优化检测性能？**

这道题在**腾讯、网易、米哈游**的客户端面试中出现频率极高，因为它直接考查"物理系统 API 选型 + 性能敏感度"。

### ✅ 参考答案

这道题有三个层次可以答：

#### 🔹 第一层：基础方案——物理引擎重叠查询

最直接的做法是用引擎的重叠检测（Overlap）API：

```cpp
// UE5 方案：SweepMultiByChannel 或 OverlapMultiByChannel
// 扇形 = 球体重叠 + 角度过滤
TArray<FOverlapResult> Overlaps;
FCollisionShape SphereShape = FCollisionShape::MakeSphere(SkillRadius);

// 第一步：球体范围粗筛
bool bHit = GetWorld()->OverlapMultiByChannel(
    Overlaps,
    SkillOrigin,
    FQuat::Identity,
    ECC_Pawn,              // 只检测 Pawn 通道
    SphereShape
);

// 第二步：扇形角度精筛
for (const FOverlapResult& Result : Overlaps)
{
    FVector ToTarget = Result.GetActor()->GetActorLocation() - SkillOrigin;
    ToTarget.Z = 0;  // 投影到水平面
    float Angle = FMath::Acos(FVector::DotProduct(
        ToTarget.GetSafeNormal(), ForwardDir));
    
    if (FMath::RadiansToDegrees(Angle) <= HalfAngle)
    {
        // 命中！加入结果列表
        HitTargets.Add(Result.GetActor());
    }
}
```

```csharp
// Unity 方案：Physics.OverlapSphereNonAlloc + 角度过滤
private Collider[] _overlapBuffer = new Collider[64]; // 预分配！

public List<GameObject> FanShapeDetect(Vector3 origin, Vector3 forward, 
    float radius, float halfAngle, LayerMask enemyLayer)
{
    List<GameObject> hits = new List<GameObject>();
    
    // 第一步：球体粗筛（NonAlloc 版本零 GC）
    int count = Physics.OverlapSphereNonAlloc(
        origin, radius, _overlapBuffer, enemyLayer);
    
    // 第二步：扇形角度精筛
    for (int i = 0; i < count; i++)
    {
        Vector3 toTarget = _overlapBuffer[i].transform.position - origin;
        toTarget.y = 0; // 投影到水平面
        float angle = Vector3.Angle(forward, toTarget);
        
        if (angle <= halfAngle)
            hits.Add(_overlapBuffer[i].gameObject);
    }
    return hits;
}
```

**核心要点**：
- UE5 的 `OverlapMulti` 底层走 Chaos 的 BroadPhase（AABB 粗筛 → NarrowPhase 精检）
- Unity **必须用 NonAlloc 版本**，`Physics.OverlapSphere` 每次调用会 new 数组，造成 GC Alloc

**优化二：空间分区预筛（纯逻辑层）**

当 Overlap 查询太频繁时，可以在**逻辑层自己维护一个网格/四叉树**做第一轮粗筛：

```lua
-- Lua 侧的简易网格空间分区
local GridSize = 500  -- 500cm 一格
local Grid = {}

function GetGridKey(pos)
    local gx = math.floor(pos.x / GridSize)
    local gy = math.floor(pos.y / GridSize)
    return gx .. "_" .. gy
end

function RegisterEnemy(enemy)
    local key = GetGridKey(enemy:GetPosition())
    Grid[key] = Grid[key] or {}
    Grid[key][enemy] = true
end

-- 技能检测时只查询技能范围覆盖的几个格子
function QueryEnemiesInRange(origin, radius)
    local candidates = {}
    local minKey = GetGridKey({x = origin.x - radius, y = origin.y - radius})
    local maxKey = GetGridKey({x = origin.x + radius, y = origin.y + radius})
    -- 遍历覆盖的格子，收集候选者
    -- 再对候选者做精确距离/角度判定
    return candidates
end
```

**优化三：时间分摊（帧预算控制）**

10 个技能不必全在同一帧检测——用帧错开策略：

```cpp
// UE5：技能检测分帧执行
void ASkillManager::Tick(float DeltaTime)
{
    constexpr int MAX_DETECTIONS_PER_FRAME = 3;
    int Processed = 0;
    
    while (!PendingDetections.IsEmpty() && Processed < MAX_DETECTIONS_PER_FRAME)
```

#### 🔹 第三层：面试加分——为什么不用射线？

面试官可能追问"为什么不用射线扫描"：

- 射线是**一条线**，检测扇形需要发射多条射线（Fan Raycast），N 条射线 = N 次查询
- 相比之下，一次 `OverlapSphere` + 角度过滤只需 **1 次物理查询**
- **但**射线适合另一个场景：**子弹命中判定**（单条射线 + 穿透判定）

| 场景 | 推荐方案 | 原因 |
|------|----------|------|
| 扇形 AOE | OverlapSphere + 角度过滤 | 1 次查询，结果完整 |
| 矩形 AOE | OverlapBox | 直接匹配形状 |
| 子弹射击 | LineTrace / Raycast | 精确命中点 + 穿透 |
| 锥形检测 | OverlapSphere + 距离衰减 | 近处角度大，远处角度小 |
| 持续伤害区域 | Trigger Volume | 事件驱动，不需要每帧查询 |

## 📖 知识讲解：游戏物理系统的"三层蛋糕"

### 碰撞检测的三阶段

无论是 UE 的 Chaos 还是 Unity 的 PhysX，碰撞检测都遵循相同的三阶段流水线：

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  Broad Phase │───▶│  Mid Phase   │───▶│ Narrow Phase │
│  AABB 粗筛   │    │  子形状筛选   │    │  精确接触计算 │
│  O(n log n)  │    │  凸包分解     │    │  GJK/EPA     │
└──────────────┘    └──────────────┘    └──────────────┘
```

**Broad Phase（粗筛）**：
- 用 AABB（轴对齐包围盒）做 Sweep-and-Prune 或空间散列
- 目标是快速排除"显然不可能碰撞"的物体对
- **客户端优化点**：碰撞层矩阵越稀疏，粗筛跳过的越多

**Mid Phase（中筛）**：
- 复合碰撞体（如角色 Capsule + 武器 Box）拆解为子形状
- 只在 Broad Phase 命中时才进入

**Narrow Phase（精筛）**：
- 用 GJK（Gilbert-Johnson-Keerthi）算法计算精确接触点和穿透深度
- 这是最耗时的阶段，碰撞体越复杂越慢

### 射线检测 API 对比

射线检测是客户端最常用的物理查询，两个引擎的 API 设计差异值得对比：

```cpp
// UE5 射线检测 —— 参数显式、功能强大
FHitResult HitResult;
FCollisionQueryParams Params;
Params.AddIgnoredActor(this);           // 忽略自身
Params.bReturnPhysicalMaterial = true;  // 返回物理材质（用于音效/特效）
Params.bTraceComplex = false;           // false=简单碰撞, true=复杂网格

bool bHit = GetWorld()->LineTraceSingleByChannel(
    HitResult,
    Start, End,
    ECC_Visibility,  // 碰撞通道
    Params
);

if (bHit)
{
    // HitResult.ImpactPoint —— 命中点（世界坐标）
    // HitResult.ImpactNormal —— 命中面法线
    // HitResult.PhysMaterial —— 物理材质（草地/金属/木头...）
    // HitResult.BoneName —— 命中骨骼名（骨骼体命中时）
}
```

```csharp
// Unity 射线检测 —— 简洁、注意 GC
RaycastHit hit;
int layerMask = LayerMask.GetMask("Enemy", "Environment");

// ✅ 推荐：用 out 参数版本，零 GC
if (Physics.Raycast(origin, direction, out hit, maxDistance, layerMask))
{
    // hit.point —— 命中点
    // hit.normal —— 命中面法线
    // hit.collider —— 命中的碰撞器
}

// ❌ 避免：Physics.RaycastAll 每次 new 数组
// ✅ 替代：Physics.RaycastNonAlloc
RaycastHit[] hitBuffer = new RaycastHit[16]; // 预分配
int hitCount = Physics.RaycastNonAlloc(origin, direction, hitBuffer, 
    maxDistance, layerMask);
```

> ⚠️ **Unity 面试高频考点**：`Physics.Raycast` 的 out 版本不产生 GC，但 `RaycastAll` 会。大量射线检测（如弹幕游戏）必须用 `NonAlloc` 系列 + 预分配缓冲区。

### UE5 Chaos vs Unity PhysX 关键差异

| 特性 | UE5 Chaos | Unity PhysX |
|------|-----------|-------------|
| 引擎 | 自研 Chaos | NVIDIA PhysX 4.x |
| 多线程 | 深度集成 TaskGraph | Job System 可加速 |
| 碰撞通道 | 自定义 Channel + Response | Layer + LayerMask |
| 异步查询 | `AsyncLineTrace` 原生支持 | 需手动 `Physics.RaycastCommand` |
| 射线返回 | `FHitResult`（含物理材质、骨骼） | `RaycastHit`（较简洁） |
| 调试工具 | `DrawDebugLine` / Chaos Visual Debugger | `Debug.DrawRay` / Physics Debugger |

## 💡 实战技巧

### 技巧一：射线检测的"三不原则"

在实际项目中，射线检测用得极多但也极容易出性能问题。记住"三不"：

1. **不在 Tick 里无脑射**——能事件驱动就不要轮询。比如"准星对准敌人高亮"，可以只在玩家转向时射，不必每帧射
2. **不用复杂碰撞**——`bTraceComplex = true` 会对 Mesh 三角面逐面检测，比简单碰撞慢 10-100 倍。只在需要精确命中部位时才开
3. **不忘设最大距离**——无限距离射线会穿越整个场景的 BroadPhase，限制 `MaxDistance` 可以大幅减少检测范围

### 技巧二：Trigger Volume vs 每帧查询

很多新手习惯在 Tick 里做 Overlap 检测伤害区域，但更优的做法是用 **Trigger Volume（触发器体积）**：

```cpp
// UE5：伤害区域用 Trigger 事件驱动
UFUNCTION()
void OnDamageZoneOverlapBegin(UPrimitiveComponent* OverlappedComp, 
    AActor* OtherActor, ...)
{
    if (IDamageable* Target = Cast<IDamageable>(OtherActor))
    {
        ActiveTargets.Add(Target);  // 进入区域，加入列表
    }
}

// Tick 里只对 ActiveTargets 列表结算伤害，不做物理查询
void Tick(float DeltaTime)
{
    for (auto* Target : ActiveTargets)
    {
        Target->ApplyDamage(DamagePerSecond * DeltaTime);
    }
}
```

> Trigger 的 Begin/End Overlap 是引擎物理步中自动计算的，不需要你额外发起查询。每帧只需遍历一个列表，比每帧 Overlap 查询高效得多。

### 技巧三：异步射线——不阻塞游戏线程

当一帧内需要大量射线（如 AI 感知系统同时检测 20 个 AI 的视线），同步射线会卡主线程。两个引擎都有异步方案：

```cpp
// UE5：AsyncLineTraceByChannel
FTraceHandle Handle = GetWorld()->AsyncLineTraceByChannel(
    EAsyncTraceType::Single,
    Start, End,
    ECC_Visibility,
    Params
);
// 下一帧通过 FTraceDelegate 回调拿结果
```

```csharp
// Unity：RaycastCommand + JobSystem 批量射线
var commands = new NativeArray<RaycastCommand>(20, Allocator.TempJob);
var results = new NativeArray<RaycastHit>(20, Allocator.TempJob);

for (int i = 0; i < 20; i++)
{
    commands[i] = new RaycastCommand(origins[i], directions[i], 
        QueryParameters.Default, maxDistance);
}

// 一次性提交 20 条射线，Job System 并行执行
JobHandle handle = RaycastCommand.ScheduleBatch(commands, results, 4);
handle.Complete();

// 读取结果...
commands.Dispose();
results.Dispose();
```

> 💡 Unity 的 `RaycastCommand` 批量提交后底层会用多线程并行执行，20 条射线的耗时接近 1 条。这是 AI 感知系统的标准做法。
