### 📖 知识讲解

#### 1. 渲染管线基础：从像素到屏幕

现代游戏渲染管线的核心流程：

```
CPU 阶段                              GPU 阶段
┌─────────────┐                     ┌──────────────────────────────────────┐
│ 场景遍历     │                     │ 顶点着色器 → 光栅化 → 片元着色器     │
│ 可见性剔除   │ ──── Draw Call ──→  │ → 深度测试 → 混合 → 帧缓冲          │
│ 排序/批处理  │                     │                                      │
└─────────────┘                     └──────────────────────────────────────┘
```

**Draw Call 是 CPU 和 GPU 之间的"指令包"**——CPU 告诉 GPU"用这个材质画这堆三角形"。Draw Call 本身的开销主要在 CPU 侧（状态切换、资源绑定），而非 GPU 计算本身。

#### 2. 前向渲染（Forward Rendering）

**核心思路**：每个物体、每个光源 → 一次（或多次）绘制。

```
对于每个物体:
    对于每个影响该物体的光源:
        执行一次渲染 Pass → 计算光照并累加
```

**时间复杂度**：O(物体数 × 光源数)

```hlsl
// 前向渲染 Shader 伪代码（单 Pass 多光源，URP 风格）
half4 ForwardFragment(Varyings input) : SV_Target
{
    half3 finalColor = 0;
    
    // 主光源（方向光）
    Light mainLight = GetMainLight(input.shadowCoord);
    finalColor += CalculateLighting(input, mainLight);
    
    // 附加光源（逐像素，URP 在单 Pass 中处理）
    int additionalLightCount = GetAdditionalLightsCount();
    for (int i = 0; i < additionalLightCount; i++)
    {
        Light light = GetAdditionalLight(i, input.positionWS);
        finalColor += CalculateLighting(input, light);
    }
    
    // 环境光 / IBL
    finalColor += CalculateAmbient(input);
    
    return half4(finalColor, 1.0);
}
```

**优点**：
- ✅ 天然支持半透明（Alpha Blending）
- ✅ 支持 MSAA 抗锯齿
- ✅ 内存带宽低（不需要 G-Buffer）
- ✅ 移动端友好（TBDR 架构优化友好）

**缺点**：
- ❌ 多光源性能线性下降（O(N×L)）
- ❌ 每个物体只能接收有限数量的光源（如 URP 默认每物体最多 8 个附加光源）

#### 3. 延迟渲染（Deferred Rendering）

**核心思路**：先把所有几何信息写入 G-Buffer，再统一做光照计算。

```
Pass 1 — 几何通道（Geometry Pass）:
    对于每个物体:
        渲染几何信息 → 写入 G-Buffer（Position, Normal, Albedo, Specular...）

Pass 2 — 光照通道（Lighting Pass）:
    对于每个光源:
        读取 G-Buffer → 计算光照贡献 → 累加到光照缓冲
```

**时间复杂度**：O(物体数 + 光源数 × 受影响像素数)

**G-Buffer 的典型布局**：

| Buffer | R | G | B | A | 格式 |
|--------|---|---|---|---|------|
| **RT0** | Albedo.r | Albedo.g | Albedo.b | AO | RGBA8 |
| **RT1** | Normal.x | Normal.y | Normal.z | Roughness | RGB10A2 / RGBA16F |
| **RT2** | Metallic | Specular | SubsurfaceProfile | ShadingModel | RGBA8 |
| **Depth** | — | — | — | — | D24S8 / D32F |

```hlsl
// G-Buffer 写入示例（几何通道）
struct GBufferOutput
{
    half4 GBuffer0 : SV_Target0;  // Albedo + AO
    half4 GBuffer1 : SV_Target1;  // WorldNormal + Roughness
    half4 GBuffer2 : SV_Target2;  // Metallic + Specular + ShadingModel
};

GBufferOutput GBufferFragment(Varyings input)
{
    GBufferOutput output;
    
    half4 albedo = tex2D(_MainTex, input.uv);
    half3 normal = normalize(input.normalWS);
    
    output.GBuffer0 = half4(albedo.rgb, _AO);
    output.GBuffer1 = half4(EncodeNormal(normal), _Roughness);
    output.GBuffer2 = half4(_Metallic, _Specular, 0, EncodeShadingModel());
    
    return output;
}
```

**优点**：
- ✅ 支持大量动态光源（成本与物体数解耦）
- ✅ 光照计算在屏幕空间，与场景复杂度无关
- ✅ 一次几何绘制 + 多次光照计算，避免重复绘制

**缺点**：
- ❌ G-Buffer 内存带宽大（移动端尤其敏感）
- ❌ 不天然支持半透明物体（需要额外的前向 Pass 处理）
- ❌ 不支持 MSAA（因为 MRT 写入），需要替代方案（如 TAA、FXAA）
- ❌ 材质多样性受限于 G-Buffer 的固定格式

#### 4. UE vs Unity 渲染架构对比

| 特性 | UE5 | Unity（URP / HDRP） |
|------|-----|---------------------|
| **默认渲染路径** | Deferred Shading | URP: Forward（可选 Deferred）；HDRP: Deferred |
| **管线可编程性** | 修改引擎源码（C++） | SRP（Scriptable Render Pipeline，C#） |
| **几何优化** | Nanite（虚拟几何体，GPU Driven） | 无内置等价方案 |
| **全局光照** | Lumen（混合 SDF + SSGI + Hardware RT） | URP: Light Probes / Lightmap；HDRP: SSGI + RT |
| **半透明处理** | 单独前向 Pass（Translucency Pass） | URP 同；HDRP 可配 |
| **移动端方案** | Mobile Forward / Forward+ | URP Forward（核心定位） |
| **材质系统** | Material Editor（节点式，HLSL） | Shader Graph + 手写 HLSL/ShaderLab |
| **Draw Call 优化** | 自动合批 + Nanite + 实例化 | SRP Batcher + GPU Instancing + Static Batching |

##### UE5 Nanite 的革命性设计

```
传统 LOD：               Nanite：
CPU 判断距离 → 选 LOD     GPU 实时判断每个三角簇（Cluster）的可见性
离散的 LOD 切换（pop）     连续 LOD，无 pop
手动创建 LOD 链            自动构建 BVH + 虚拟几何体
Draw Call 数 ≈ 物体数      Draw Call 数 ≈ 常数（GPU Driven）
```

Nanite 本质上将 **Visibility Buffer** 的思想推到了极致——用 GPU 来判断哪些三角形/像素是可见的，跳过了传统的 CPU 场景遍历。

##### Unity SRP Batcher 的优化思路

```csharp
// Unity SRP Batcher 优化原理：
// 传统：每个 Draw Call 都需要设置完整的渲染状态
// SRP Batcher：将 Shader 属性和材质属性分离

// 1. Shader 变体不变 → 不需要重新绑定 Shader
// 2. 材质属性存入 Persistent CBUFFER → 减少 CPU 设置开销
// 3. 按 Shader 排序 → 最小化状态切换

// 要触发 SRP Batcher，Shader 需要遵循 CBUFFER 规则：
CBUFFER_START(UnityPerMaterial)
    half4 _BaseColor;
    half _Metallic;
    half _Smoothness;
CBUFFER_END
```

### 💡 实战技巧

**1. Draw Call 优化不是越少越好**

Draw Call 优化的目标是**减少 CPU 瓶颈**，但过度合批可能导致：
- GPU 过度绘制（Overdraw）增加，因为合批后更难做前后排序
- 大 Mesh 无法被视锥剔除（Frustum Culling），浪费 GPU 算力

最佳实践：先用 RenderDoc / Nsight / Frame Debugger 找到真正的瓶颈，再针对性优化。

**2. UE5 中 Nanite 不适用的场景**

Nanite 目前（UE 5.4+）**不支持**：
- 半透明材质
- 骨骼动画 Mesh（蒙皮网格）
- 部分特殊材质（如世界位置偏移 WPO）

所以角色、特效、植被（带风力摆动的）仍需传统 LOD + Instancing 方案。

**3. Unity 中 URP 和 HDRP 不能混用**

这是常见误区。SRP 是管线级别的选择：
- **URP**：定位中低端 + 跨平台，功能取舍后性能优先
- **HDRP**：定位高端 PC/主机，功能最全但不支持移动端
- **不能在同一个项目中混用两个 SRP**，需要在项目初期确定

**4. Lua 侧关注渲染的什么？**

作为 Lua 层的逻辑开发者，虽然不直接写 Shader，但需要关注：
- **材质实例动态参数设置**（`SetScalarParameterValue`、`SetVectorParameterValue`）
- **渲染状态控制**（隐藏/显示、LOD 强制切换）
- **粒子和特效的性能预算**（Overdraw 是移动端杀手）
- **UI 渲染与场景渲染的混合**（避免全屏 UI 触发不必要的场景渲染）

```lua
-- Lua 侧控制材质参数示例（UnLua / sluaunreal 风格）
function SetWeatherEffect(meshComp, rainIntensity)
    if not meshComp then return end
    
    -- 动态修改材质参数（会创建 Material Instance Dynamic）
    local matInst = meshComp:CreateDynamicMaterialInstance(0)
    if matInst then
        matInst:SetScalarParameterValue("RainIntensity", rainIntensity)
        matInst:SetScalarParameterValue("WetnessFactor", 
            math.min(rainIntensity * 1.5, 1.0))
    end
end

-- 注意：频繁创建 DynamicMaterialInstance 是性能陷阱！
-- 应该缓存 matInst，只在第一次调用时创建
local cachedMaterials = {}
function GetOrCreateMaterial(meshComp, slotIndex)
    local key = tostring(meshComp) .. "_" .. slotIndex
    if not cachedMaterials[key] then
        cachedMaterials[key] = meshComp:CreateDynamicMaterialInstance(slotIndex)
    end
    return cachedMaterials[key]
end
```

### 📖 知识讲解

#### 1. 智能指针家族速览

| 指针类型 | 语义 | 引用计数 | 循环引用 | 线程安全 | UE4 等价 |
|---------|------|---------|---------|---------|---------|
| `std::unique_ptr` | 独占所有权 | 无 | N/A | N/A | `TUniquePtr` |
| `std::shared_ptr` | 共享所有权 | 有（强引用） | ❌ 会 | 引用计数操作原子 | `TSharedPtr` |
| `std::weak_ptr` | 观察者（不拥有） | 有（弱引用） | ✅ 打破 | 同 shared | `TWeakPtr` |

#### 2. `shared_ptr` 底层实现：控制块（Control Block）

```cpp
// shared_ptr 的简化内部结构
template<typename T>
class shared_ptr
{
    T* _ptr;                    // 指向实际对象
    ControlBlock* _control;     // 指向控制块
};

struct ControlBlock
{
    std::atomic<int> strong_count;  // 强引用计数
    std::atomic<int> weak_count;    // 弱引用计数（weak_ptr 用）
    // Deleter、Allocator 等...
};
```

**关键细节**：
- `strong_count` 归零时 → 销毁对象（调用析构函数 + 释放对象内存）
- `weak_count` 也归零时 → 释放控制块本身的内存
- 引用计数操作是 **原子的**（`std::atomic`），但 **shared_ptr 对象本身不是线程安全的**

```cpp
// ⚠️ 常见误解：shared_ptr 的"线程安全"

// ✅ 安全：多线程中对不同 shared_ptr 副本的操作
std::shared_ptr<GameData> globalData = std::make_shared<GameData>();

// Thread A:
auto localA = globalData;  // 拷贝，原子增加引用计数 ✅

// Thread B:
auto localB = globalData;  // 拷贝，原子增加引用计数 ✅

// ❌ 不安全：多线程中对同一个 shared_ptr 实例的读写
// Thread A:
globalData = newData;      // 修改 globalData ❌

// Thread B:
auto localB = globalData;  // 读取 globalData ❌（数据竞争！）

// ✅ 正确做法：用 mutex 保护或用 std::atomic<std::shared_ptr>（C++20）
std::mutex dataMutex;
// Thread A:
{
    std::lock_guard<std::mutex> lock(dataMutex);
    globalData = newData;
}
```

#### 3. UE4 的 TSharedPtr：为什么有两种线程模式？

```cpp
// UE4 的智能指针有两种模式
TSharedPtr<FMyStruct, ESPMode::NotThreadSafe>  // 非线程安全（默认）
TSharedPtr<FMyStruct, ESPMode::ThreadSafe>      // 线程安全

// 为什么不默认都用 ThreadSafe？
// 因为原子操作有性能开销！
```

**性能对比**（典型场景，x86-64）：

| 操作 | 非原子 (NotThreadSafe) | 原子 (ThreadSafe) | 差异 |
|------|----------------------|-------------------|------|
| 引用计数递增 | ~1 ns | ~5-15 ns | 5-15x |
| 引用计数递减 | ~1 ns | ~5-15 ns | 5-15x |
| 拷贝 shared_ptr | ~2 ns | ~10-30 ns | 5-15x |

在游戏引擎中，一帧可能有 **数万次** 智能指针拷贝/销毁操作。如果所有智能指针都用原子操作，累积开销相当可观。因此 UE4 默认使用 `NotThreadSafe` 模式，只在确实需要跨线程共享时才切换为 `ThreadSafe`。

```cpp
// UE4 实际使用示例

// GameThread 内部使用（不跨线程）→ NotThreadSafe（默认）
TSharedPtr<FWidgetData> UIData = MakeShared<FWidgetData>();

// 需要跨线程传递 → ThreadSafe
TSharedPtr<FAsyncResult, ESPMode::ThreadSafe> AsyncResult = 
    MakeShared<FAsyncResult, ESPMode::ThreadSafe>();

// 在工作线程中安全访问
Async(EAsyncExecution::ThreadPool, [AsyncResult]()
{
    AsyncResult->Data = ComputeHeavyStuff();
    AsyncResult->bComplete = true;
});
```

#### 4. 移动语义：从"昂贵的拷贝"到"廉价的转移"

**问题场景**：游戏中大量临时数据需要传递

```cpp
// ❌ 没有移动语义的世界（C++03）
std::vector<FVertex> GenerateMeshVertices()
{
    std::vector<FVertex> Vertices;
    Vertices.reserve(10000);
    // ... 填充 10000 个顶点
    return Vertices;  // 拷贝 10000 个顶点到调用方！（虽然有 NRVO 优化但不保证）
}

// 如果要传给另一个函数：
void ProcessMesh(std::vector<FVertex> Verts)  // 又一次拷贝！
{
    // ...
}

auto verts = GenerateMeshVertices();  // 拷贝 1（NRVO 可能优化）
ProcessMesh(verts);                    // 拷贝 2（10000 个顶点的深拷贝）
```

**移动语义的本质**：**转移资源的所有权，而非复制资源本身。**

```cpp
// ✅ 移动语义的世界（C++11+）

// 移动构造函数：偷走源对象的资源
class MeshData
{
    FVertex* Vertices = nullptr;
    int32 Count = 0;

public:
    // 拷贝构造（昂贵）
    MeshData(const MeshData& Other)
        : Count(Other.Count)
    {
        Vertices = new FVertex[Count];
        FMemory::Memcpy(Vertices, Other.Vertices, Count * sizeof(FVertex));
        // O(N) — 实际复制所有数据
    }

    // 移动构造（廉价）
    MeshData(MeshData&& Other) noexcept
        : Vertices(Other.Vertices)  // 偷指针
        , Count(Other.Count)
    {
        Other.Vertices = nullptr;   // 让源对象放弃所有权
        Other.Count = 0;
        // O(1) — 只是交换了两个指针/整数
    }
    
    ~MeshData()
    {
        delete[] Vertices;
    }
};
```

**图解移动 vs 拷贝**：

```
拷贝（深拷贝）：
src:  [ptr] ──→ [████████████████]  (10000 vertices)
              ↓ 分配新内存 + 逐字节复制
dst:  [ptr] ──→ [████████████████]  (10000 vertices 的副本)

移动（所有权转移）：
src:  [ptr] ──→ [████████████████]  (10000 vertices)
              ↓ 偷走指针
dst:  [ptr] ──→ [████████████████]  (同一块内存)
src:  [nullptr]  (已失效，但仍然是有效对象)
```

#### 5. `std::move` 不"移动"——它只是一个 cast

```cpp
// std::move 的真实实现（简化版）
template<typename T>
constexpr std::remove_reference_t<T>&& move(T&& arg) noexcept
{
    return static_cast<std::remove_reference_t<T>&&>(arg);
}

// 它只是把左值（lvalue）转换成右值引用（rvalue reference）
// 真正的"移动"发生在 移动构造函数 / 移动赋值运算符 被调用时

std::vector<int> a = {1, 2, 3, 4, 5};
std::vector<int> b = std::move(a);  // std::move 只是 cast
                                     // vector 的移动构造函数才真正移动了数据

// 移动之后，a 处于"有效但未指定"的状态
// a.size() 可能是 0（大多数实现），但标准只保证 a 可以安全析构
```

#### 6. 游戏开发中的移动语义实战

```cpp
// === 场景 1：容器中高效插入大对象 ===

struct FParticleEmitter
{
    TArray<FVector> Positions;      // 可能有上万个粒子位置
    TArray<FLinearColor> Colors;    // 对应颜色
    UMaterialInterface* Material;
    
    // 支持移动
    FParticleEmitter(FParticleEmitter&& Other) noexcept = default;
    FParticleEmitter& operator=(FParticleEmitter&& Other) noexcept = default;
};

TArray<FParticleEmitter> ActiveEmitters;

void AddEmitter(FParticleEmitter NewEmitter)
{
    // 如果调用方传入的是临时对象或 std::move 后的对象，
    // 这里会触发移动构造而非拷贝构造
    ActiveEmitters.Emplace(MoveTemp(NewEmitter));  // UE4 用 MoveTemp 替代 std::move
}

// === 场景 2：函数返回大对象 ===

TArray<FHitResult> PerformBatchRaycast(const TArray<FRay>& Rays)
{
    TArray<FHitResult> Results;
    Results.Reserve(Rays.Num());
    
    for (const auto& Ray : Rays)
    {
        FHitResult Hit;
        if (GetWorld()->LineTraceSingleByChannel(Hit, Ray.Origin, 
            Ray.Origin + Ray.Direction * MaxDistance, ECC_Visibility))
        {
            Results.Add(Hit);
        }
    }
    
    return Results;  // NRVO（命名返回值优化）或移动语义
                      // 不会发生昂贵的拷贝
}

// === 场景 3：unique_ptr 在游戏系统中的使用 ===

class FGameStateManager
{
    // 游戏状态独占拥有，不共享
    TUniquePtr<FGameState> CurrentState;
    
public:
    void TransitionTo(TUniquePtr<FGameState> NewState)
    {
        if (CurrentState)
        {
            CurrentState->OnExit();
        }
        CurrentState = MoveTemp(NewState);  // 移动所有权
        CurrentState->OnEnter();
    }
};

// 调用方必须转移所有权
auto MenuState = MakeUnique<FMenuState>();
StateManager.TransitionTo(MoveTemp(MenuState));
// 此时 MenuState 已为 nullptr
```

### 💡 实战技巧

**1. UE4 中 MoveTemp vs std::move**

UE4 推荐使用 `MoveTemp()` 而非 `std::move()`，因为 `MoveTemp` 在 Debug 构建中会进行额外的**静态断言**，确保你不会对 const 对象调用移动：

```cpp
// MoveTemp 的简化实现
template<typename T>
FORCEINLINE typename TRemoveReference<T>::Type&& MoveTemp(T&& Obj)
{
    typedef typename TRemoveReference<T>::Type CastType;
    
    // Debug 断言：不能移动 const 对象
    static_assert(TIsLValueReferenceType<T>::Value, "MoveTemp called on rvalue");
    static_assert(!TAreTypesEqual<CastType&, const CastType&>::Value, 
                  "MoveTemp called on const object");
    
    return (CastType&&)Obj;
}
```

**2. 返回值优化（RVO/NRVO）优先于 std::move**

```cpp
// ❌ 不要这样做
TArray<int32> GetData()
{
    TArray<int32> Result;
    // ...
    return std::move(Result);  // 反模式！会阻止 NRVO！
}

// ✅ 正确：直接 return
TArray<int32> GetData()
{
    TArray<int32> Result;
    // ...
    return Result;  // 编译器会自动应用 NRVO（零拷贝零移动）
}
```

**3. Lua 中的值语义 vs C++ 的所有权**

Lua 是 GC 语言，table 变量本身就是引用。与 C++ 交互时需要注意：

```lua
-- Lua 侧：table 赋值是引用（浅拷贝）
local original = { hp = 100, position = {x=0, y=0, z=0} }
local ref = original       -- ref 和 original 指向同一个 table
ref.hp = 50                -- original.hp 也变成 50 了！

-- 如果需要深拷贝（类似 C++ 的拷贝语义）
function DeepCopy(obj)
    if type(obj) ~= "table" then return obj end
    local copy = {}
    for k, v in pairs(obj) do
        copy[DeepCopy(k)] = DeepCopy(v)
    end
    return setmetatable(copy, getmetatable(obj))
end

local independent = DeepCopy(original)  -- 独立副本
independent.hp = 50                      -- 不影响 original
```

**4. 避免在热路径上使用 shared_ptr**

```cpp
// ❌ 性能敏感的每帧更新中频繁拷贝 shared_ptr
void TickAllEntities(const TArray<TSharedPtr<FEntity>>& Entities)
{
    for (auto Entity : Entities)  // 每次循环都拷贝一次 shared_ptr（引用计数 +1/-1）
    {
        Entity->Tick(DeltaTime);
    }
}

// ✅ 使用 const 引用避免不必要的引用计数操作
void TickAllEntities(const TArray<TSharedPtr<FEntity>>& Entities)
{
    for (const auto& Entity : Entities)  // const& 不增加引用计数
    {
        Entity->Tick(DeltaTime);
    }
}
```
