## 📝 上期思考题回顾

> **第 9 期思考题**：在一个开放世界游戏中，玩家骑马在地图上奔跑，周围不断加载/卸载大量资源。请设计一个**资源引用计数 + 延迟卸载**的方案：何时增加引用？何时减少？引用归零后是否立刻卸载？如何避免"刚卸载又立刻加载回来"的抖动？

### ✅ 参考答案

核心方案：**引用计数 + 冷却队列 + 距离预测**，三管齐下。

```cpp
// 简化版资源管理器（UE5 风格伪代码）
struct FManagedAsset
{
    int32 RefCount = 0;       // 引用计数
    float UnusedTimer = 0.f;  // 归零后已等待时长
    float CooldownSec = 5.f;  // 冷却时间（不同资源类型不同）
    TSoftObjectPtr<UObject> Asset;
};

// 加载 —— 谁用谁 AddRef
void AddRef(FName AssetId)
{
    auto& Info = AssetMap[AssetId];
    Info.RefCount++;
    Info.UnusedTimer = 0.f;  // 重新激活，取消冷却
    if (!Info.Asset.IsValid())
        AsyncLoad(AssetId);  // 首次引用触发异步加载
}

// 卸载 —— 用完 Release，但不立即销毁
void Release(FName AssetId)
{
    auto& Info = AssetMap[AssetId];
    Info.RefCount = FMath::Max(0, Info.RefCount - 1);
    // RefCount 归零后不立刻卸载，进入冷却队列
}

// 每帧 Tick —— 冷却到期才真正卸载
void Tick(float DeltaTime)
{
    for (auto& [Id, Info] : AssetMap)
    {
        if (Info.RefCount == 0)
        {
            Info.UnusedTimer += DeltaTime;
            if (Info.UnusedTimer >= Info.CooldownSec)
                RealUnload(Id);  // 真正释放
        }
    }
}
```

**防抖动关键**：
1. **冷却时间分级**：高频出现的资源（地形 Tile、常见植被）冷却 8-10 秒，低频资源（远处 NPC）冷却 3 秒
2. **距离预测**：玩家正朝某方向移动时，该方向上刚卸载的资源不进入冷却队列，直接保留
3. **LRU 兜底**：内存超预算时，按"最久未使用"强制卸载冷却队列中的资源

> 这个思路和今天要讲的 **Addressables 引用计数** 一脉相承。

## 🎯 面试题

**Q：请对比 UE5 的 AssetManager 和 Unity 的 Addressables，分别是怎么管理运行时资源的？各自的引用/释放机制有什么区别？** ⭐⭐ 中级

### 📋 参考答案

|  | **UE5 AssetManager** | **Unity Addressables** |
|---|---|---|
| **核心理念** | "主资源"分类管理 + 软引用异步加载 | "地址标签"寻址 + 引用计数自动管理 |
| **资源标识** | `FSoftObjectPath` / `TSoftObjectPtr` | `AssetReference` / 字符串地址 |
| **加载方式** | `AsyncLoadPrimaryAsset()` / `StreamableManager` | `Addressables.LoadAssetAsync<T>()` |
| **释放方式** | 手动 `Unload()` / `ReleaseHandle` | `Addressables.Release(handle)` 引用计数归零自动卸载 |
| **依赖处理** | 引擎内部 Package 依赖自动解析 | Bundle 依赖自动追踪，引用计数传递 |
| **热更新** | 自带 Pak/Chunk 机制，需手动管理 | 内置 Remote Catalog + CRC 校验 |
| **典型坑** | 软引用忘记加载就访问 → 崩溃 | Release 少调一次 → 内存泄漏 |

**一句话总结**：UE 更"手动挡"，灵活但需要自己管好生命周期；Unity Addressables 更"自动挡"，引用计数帮你管，但你必须**严格配对 Load/Release**。

## 📖 知识讲解

### 1. UE5 的资源管理体系

UE5 的资源管理有三层，面试时搞清楚层级关系就赢了一半：

```
硬引用 (Hard Reference)  ← 打包时一起加载，简单但内存浪费
软引用 (Soft Reference)  ← 只存路径，运行时按需异步加载 ✅ 推荐
AssetManager            ← 在软引用基础上的上层管理框架
```

**面试最爱考的：软引用 + 异步加载**

```cpp
// UE5 —— 用 TSoftObjectPtr 做软引用
UPROPERTY(EditDefaultsOnly)
TSoftObjectPtr<UTexture2D> IconTexture;

// 异步加载（不阻塞游戏主线程）
FStreamableManager& Loader = UAssetManager::GetStreamableManager();
Loader.RequestAsyncLoad(
    IconTexture.ToSoftObjectPath(),
    FStreamableDelegate::CreateLambda([this]()
    {
        // 加载完成回调 —— 在这里安全使用资源
        UTexture2D* Tex = IconTexture.Get();
        if (Tex) SetIcon(Tex);
    })
);
```

**核心要点**：
- `TSoftObjectPtr` 编辑器里可以选资源，但**运行时不会自动加载**
- 用 `StreamableManager` 或 `AssetManager` 发起异步加载
- 加载完后通过 `.Get()` 拿到真正的对象指针

### 2. Unity Addressables 的资源管理体系

Addressables 的核心卖点：**你不用关心资源在哪个 Bundle 里，只需要知道它的"地址"**。

```csharp
// Unity C# —— Addressables 异步加载
using UnityEngine.AddressableAssets;
using UnityEngine.ResourceManagement.AsyncOperations;

public class WeaponLoader : MonoBehaviour
{
    [SerializeField]
    private AssetReference weaponPrefabRef;  // 编辑器拖入

    private AsyncOperationHandle<GameObject> _handle;

    public async void LoadWeapon()
    {
        _handle = weaponPrefabRef.LoadAssetAsync<GameObject>();
        await _handle.Task;

        if (_handle.Status == AsyncOperationStatus.Succeeded)
            Instantiate(_handle.Result);
    }

    // ⚠️ 关键：用完必须 Release！
    private void OnDestroy()
    {
        Addressables.Release(_handle);
    }
}
```

**引用计数机制**（面试高频）：

```
LoadAssetAsync()   → RefCount +1
Release(handle)    → RefCount -1
RefCount == 0      → 自动卸载资源及所在 Bundle
```

⚠️ **最经典的坑**：`Instantiate` 出来的 GameObject **不会自动增加引用计数**！你必须手动管理：

```csharp
// ❌ 错误：Release 了 handle 但 Instantiate 的对象还在用
Addressables.Release(handle);  // Bundle 被卸载 → 材质丢失变紫色

// ✅ 正确：先销毁实例，再 Release
Destroy(weaponInstance);
Addressables.Release(handle);
```

### 3. Lua 层的资源管理（实战补充）

在很多项目里，Lua 做业务逻辑，资源加载也在 Lua 层发起：

```lua
-- Lua 层封装异步加载（常见项目模式）
local ResManager = {}
local _cache = {}  -- { [path] = { asset=obj, refCount=n } }

function ResManager:LoadAsync(path, callback)
    if _cache[path] then
        _cache[path].refCount = _cache[path].refCount + 1
        callback(_cache[path].asset)
        return
    end
    -- 调用引擎侧异步加载
    CS.ResourceLoader.LoadAsync(path, function(asset)
        _cache[path] = { asset = asset, refCount = 1 }
        callback(asset)
    end)
end

function ResManager:Release(path)
    local info = _cache[path]
    if not info then return end
    info.refCount = info.refCount - 1
    if info.refCount <= 0 then
        -- 延迟卸载：放入待清理队列而非立即卸载
        _pendingUnload[path] = os.clock()
        -- 真正卸载在定时器里做（防抖动）
    end
end
```

> 面试 Tip：如果项目用 Lua 做 UI 资源管理，面试官很可能追问"Lua 层怎么防泄漏"。答案是**严格的 AddRef/Release 配对 + 场景切换时的全量泄漏检测**。

## 💡 实战技巧

### 技巧 1：资源审计——上线前必做

不管 UE 还是 Unity，上线前一定要做**资源引用审计**：

| 检查项 | UE5 做法 | Unity 做法 |
|---|---|---|
| 找硬引用 | Reference Viewer（右键资源 → Size Map） | Addressables Analyze 窗口 |
| 内存快照 | `memreport -full` 控制台命令 | Memory Profiler 包 |
| 泄漏检测 | `obj list` 查看未释放对象 | Addressables Event Viewer 看引用计数 |

### 技巧 2：避免"启动全家桶"

新手常犯的错——主角蓝图硬引用了武器、武器硬引用了特效、特效硬引用了材质……一启动就把半个项目加载进内存。

**解决套路**：
1. **数据表驱动**：武器/技能/特效的资源路径存配置表，运行时按需加载
2. **软引用替代硬引用**：`TSoftObjectPtr`（UE）/ `AssetReference`（Unity）
3. **分层加载**：登录 → 只加载 UI 资源；进战斗 → 才加载角色/场景资源

### 技巧 3：Bundle 分包策略（Unity 特有高频题）

```
推荐分包原则：
├── 按场景分：每个场景一个 Group → 场景切换时整包卸载
├── 按类型分：Texture / Audio / Prefab 各自一个 Group → 大小可控
├── 共享资源单独分：公共材质/Shader 单独 Group → 避免重复打包
└── 热更资源分开：需要热更的 Remote，不更的 Local
```
